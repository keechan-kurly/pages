문제 발견 — 높이가 왜 이전 값 그대로일까

검색 결과 화면을 Compose로 마이그레이션하던 중이었습니다. RecyclerView의 ViewHolder 안에 ComposeView를 넣고 setContent로 Composable을 바인딩하는, Compose 마이그레이션 과도기에서 흔히 쓰는 패턴입니다.

그런데 스크롤을 올렸다 내리면 상품 카드의 높이가 이전 아이템의 높이로 고정된 채 바뀌지 않는 현상이 발생했습니다. 다른 데이터가 바인딩되어 내용은 바뀌는데, 높이만 이전 ViewHolder의 값을 유지하는 것이었습니다.

재현 조건이 흥미로웠습니다. GridLayoutManager에서 spanCount가 2 이상일 때만 발생하고, LinearLayoutManager이거나 spanCount가 1이면 정상 동작했습니다.

Google Issue Tracker를 검색해보니 동일한 문제가 이미 보고되어 있었습니다.[1][2] 하지만 Google 측의 답변은 "별도 수정 계획 없음"이었습니다. 직접 원인을 파악하고 해결해야 했습니다.

높이 변화 로그 추적

문제를 구체적으로 확인하기 위해, 재활용되는 두 ViewHolder의 height와 measuredHeight를 추적했습니다.

ComposeView@52685 child.height = 1127 child.measuredHeight = 1070 ← 올바른 높이를 측정했지만... ComposeView@52702 child.height = 1127 child.measuredHeight = 1127 ← 이 ViewHolder는 아직 갱신 전 ComposeView@52685 child.height = 1127 ← 결국 height는 갱신되지 않음 child.measuredHeight = 1127 ← measuredHeight도 원래 값으로 복귀

한 ViewHolder가 올바른 measuredHeight(1070)를 계산했지만, 최종 height는 갱신되지 않았습니다. GridLayoutManager는 같은 행의 ViewHolder 중 최대 높이를 기준으로 모든 ViewHolder의 높이를 설정하는데, 나머지 ViewHolder의 높이가 재설정되지 않아 자연스럽게 이전 최대값을 따라가는 것이었습니다.

이 동작을 이해하려면, 먼저 Android가 View의 크기를 언제 다시 측정하는지를 알아야 합니다.

배경지식: invalidate()와 requestLayout()

Android의 View 렌더링은 measure → layout → draw 세 단계로 이루어집니다. 이 파이프라인에서 View가 변경을 알리는 방법은 두 가지입니다.

requestLayout()은 "내 크기가 바뀌었으니 다시 측정해달라"는 요청입니다. 부모 View까지 전파되어 measure → layout → draw 전체 파이프라인이 재실행됩니다. 반면 invalidate()는 "다시 그려달라"는 요청으로, draw만 재실행합니다. 크기나 위치는 변하지 않습니다.

핵심 차이

ComposeView 안의 콘텐츠가 바뀌어 높이가 달라져야 할 때, requestLayout()이 호출되어야 부모(RecyclerView)가 새로운 높이를 인식합니다. invalidate()만 호출되면 내부적으로 다시 그리기만 하고, 부모는 이전 높이를 그대로 유지합니다.

이 차이가 문제의 핵심입니다. Compose 내부에서 어떤 조건일 때 requestLayout() 대신 invalidate()를 호출하는지 추적해야 합니다.

ComposeView의 측정 플로우

ComposeView의 내부 구조를 먼저 살펴보겠습니다. ComposeView는 View 계층과 Compose 계층을 연결하는 브릿지 역할의 ViewGroup이고, 그 안에 실제 Compose UI를 렌더링하는 AndroidComposeView가 자식으로 존재합니다.

재활용된 ViewHolder에서 composeView.setContent { ... }이 호출되면, 아래와 같은 순서로 크기 측정이 이루어집니다.

setContent 호출 시 측정 순서
  1. ComposeView.onMeasure() — 부모(RecyclerView)로부터 MeasureSpec을 전달받아 측정 시작
  2. AndroidComposeView.onMeasure() — MeasureSpec을 Compose의 Constraints로 변환
  3. ComposeView.Content() — 새 Composable 콘텐츠가 Composition에 등록
  4. AndroidComposeView.scheduleMeasureAndLayout() — 크기 재측정 예약

여기서 주목할 점은, 2단계에서 AndroidComposeView.onMeasure()가 실행될 때 아직 3단계의 Content()가 호출되지 않았다는 것입니다. 즉, 새 콘텐츠가 적용되기 전에 크기를 먼저 측정합니다. 이 때문에 4단계에서 재측정을 예약해야 하는데, 바로 이 4단계에서 문제가 발생합니다.

원인: scheduleMeasureAndLayout()의 분기

AndroidComposeView의 scheduleMeasureAndLayout() 함수를 살펴보겠습니다. 이 함수가 requestLayout()과 invalidate() 중 어느 것을 호출할지 결정합니다.

// AndroidComposeView.android.kt
private fun scheduleMeasureAndLayout(
    nodeToRemeasure: LayoutNode? = null
) {
    if (!isLayoutRequested && isAttachedToWindow) {
        if (nodeToRemeasure != null) {
            var node = nodeToRemeasure
            while (node != null &&
                node.measuredByParent == UsageByParent.InMeasureBlock &&
                node.childSizeCanAffectParentSize()
            ) {
                node = node.parent
            }
            if (node === root) {
                requestLayout()  // ← 이 경로로 가야 높이 갱신됨
                return
            }
        }
        if (width == 0 || height == 0) {
            requestLayout()
        } else {
            invalidate()  // ← 여기로 빠지면 높이 갱신 안 됨
        }
    }
}

while 루프는 현재 LayoutNode에서 root까지 올라가면서, 자식 크기 변화가 부모 크기에 영향을 줄 수 있는지를 확인합니다. root까지 도달하면 requestLayout()을 호출합니다. 하지만 중간에 childSizeCanAffectParentSize()가 false를 반환하면 루프를 빠져나와 invalidate()만 호출합니다.

childSizeCanAffectParentSize()는 무엇을 검사하는가

// AndroidComposeView.android.kt
private fun LayoutNode.childSizeCanAffectParentSize(): Boolean {
    return wasMeasuredWithMultipleConstraints ||
        parent?.hasFixedInnerContentConstraints == false
}

// LayoutNode.kt
internal val hasFixedInnerContentConstraints: Boolean
    get() {
        val innerContentConstraints =
            innerCoordinator.lastMeasurementConstraints
        return innerContentConstraints.hasFixedWidth
            && innerContentConstraints.hasFixedHeight
    }

hasFixedInnerContentConstraints는 부모 LayoutNode의 제약(Constraints)이 너비와 높이 모두 고정인지를 확인합니다. 고정 제약이란 minWidth == maxWidth이고 minHeight == maxHeight인 경우, 즉 MeasureSpec이 EXACTLY인 상황입니다.

부모의 제약이 고정이면, 자식 크기가 바뀌어도 부모 크기에 영향을 줄 수 없으므로 childSizeCanAffectParentSize()는 false를 반환합니다. while 루프가 root에 도달하지 못하고, 결국 invalidate()만 호출됩니다.

핵심 메커니즘: Compose는 "부모 제약이 고정이면 자식 크기 변경이 부모에 영향 없다"고 판단하여 requestLayout()을 생략합니다. 이 최적화가 RecyclerView + GridLayoutManager 환경에서는 오작동합니다.

GridLayoutManager vs LinearLayoutManager

그렇다면 왜 GridLayoutManager에서만 문제가 발생할까요? 답은 MeasureSpec에 있습니다.

LayoutManager Width MeasureSpec Height MeasureSpec 문제 발생
LinearLayoutManager EXACTLY (match_parent) UNSPECIFIED 정상
GridLayoutManager (span=1) EXACTLY UNSPECIFIED 정상
GridLayoutManager (span≥2) EXACTLY EXACTLY / AT_MOST 높이 고정

GridLayoutManager에서 spanCount가 2 이상이면, 같은 행에 여러 아이템이 배치됩니다. 이때 GridLayoutManager는 행 내 최대 높이를 기준으로 모든 아이템의 높이를 맞추기 위해, 높이 방향 MeasureSpec을 EXACTLY 또는 AT_MOST로 전달합니다.

반면 LinearLayoutManager이거나 spanCount가 1이면, 한 줄에 하나의 아이템만 있으므로 높이 제약이 없어 UNSPECIFIED로 전달합니다.

MeasureSpec이 EXACTLY일 때, AndroidComposeView.onMeasure()에서 updateRootConstraints()를 통해 갱신되는 Constraints의 너비와 높이가 모두 고정값이 됩니다. 이 고정 Constraints가 hasFixedInnerContentConstraints를 true로 만들고, 결과적으로 scheduleMeasureAndLayout()이 requestLayout() 대신 invalidate()를 호출하게 됩니다.

Compose의 입장에서 이 최적화는 합리적입니다. 부모가 고정 크기를 지정했다면, 자식이 아무리 크기를 바꿔도 부모 크기에 영향이 없으니 requestLayout()이 불필요하다고 판단한 것입니다. 하지만 RecyclerView + GridLayoutManager 환경에서는, ViewHolder가 재활용되면서 새 콘텐츠의 높이가 이전과 다를 수 있다는 상황을 고려하지 못합니다.

해결: 한 줄의 requestLayout()

원인을 파악했으니 해결은 간단합니다. setContent 호출 후 AndroidComposeView에 명시적으로 requestLayout()을 호출하면 됩니다.

ComposeView.getChildAt(0)이 바로 AndroidComposeView입니다. ComposeView의 유일한 자식이므로 인덱스 0으로 접근할 수 있습니다.

Before
fun bind(item: UIModel) {
    composeView.setContent {
        ProductScreen(item = item)
    }
    // setContent만으로는 GridLayout에서
    // 높이가 갱신되지 않음
}
After
fun bind(item: UIModel) {
    composeView.setContent {
        ProductScreen(item = item)
    }
    composeView.getChildAt(0)
        ?.requestLayout()
}

setContent 이후에 requestLayout()을 명시적으로 호출하면, Compose 내부의 scheduleMeasureAndLayout() 분기 로직과 무관하게 Android View 시스템의 표준 재측정 경로를 강제로 실행합니다. 부모인 ComposeView, 그리고 그 부모인 RecyclerView까지 크기 변경이 전파되어 높이가 정상적으로 갱신됩니다.

실제 적용 코드

class SearchProductViewHolder(
    private val composeView: ComposeView,
) : RecyclerView.ViewHolder(composeView) {

    fun bind(item: SectionProductListItemUIModel) {
        composeView.setContent {
            ProductScreen(
                item = item,
                sectionPosition = sectionPosition,
                productPosition = productPosition ?: Int.MIN_VALUE
            )
        }

        /**
         * GridLayoutManager + ComposeView 조합으로 구현 시
         * ViewHolder 높이가 갱신되지 않는 이슈로 인해 추가
         */
        composeView.getChildAt(0)?.requestLayout()
    }
}

정리

이번 이슈를 통해 Android View 시스템과 Compose 내부 측정 메커니즘의 상호작용을 깊이 이해할 수 있었습니다.

핵심 요약
  • 증상: GridLayoutManager(span≥2) + ComposeView 조합에서 ViewHolder 재활용 시 높이가 이전 값으로 고정
  • 원인: GridLayoutManager가 전달하는 EXACTLY MeasureSpec으로 인해 Compose의 scheduleMeasureAndLayout()이 requestLayout() 대신 invalidate()를 호출
  • 해결: setContent 후 composeView.getChildAt(0)?.requestLayout() 한 줄 추가

비슷한 상황에서의 적용 가이드

RecyclerView + ComposeView 하이브리드 패턴을 사용하는 경우, 아래 조건이 모두 해당되면 동일한 해결 방법을 적용하는 것을 권장합니다.

  • GridLayoutManager에서 spanCount가 2 이상인 경우
  • ViewHolder가 재활용되면서 다른 높이의 콘텐츠가 바인딩되는 경우
  • ViewHolder 내부에 ComposeView를 사용하는 경우

반면 LinearLayoutManager를 사용하거나, StaggeredGridLayoutManager처럼 아이템마다 독립적인 높이를 허용하는 경우에는 이 문제가 발생하지 않습니다. MeasureSpec이 UNSPECIFIED로 전달되어 Compose의 scheduleMeasureAndLayout()이 requestLayout()을 정상 호출하기 때문입니다.

이 문제는 View 시스템과 Compose의 하이브리드 환경에서 발생하는 과도기적 이슈입니다. 전체 화면을 Compose로 마이그레이션하면 자연스럽게 해소되지만, 마이그레이션 과정에서 RecyclerView를 유지해야 하는 상황이라면 기억해둘 만한 해결 방법입니다.
