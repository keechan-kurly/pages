Impression 이벤트 전송 구조 개선

Strategy 패턴 기반 스크롤/노출 정책 분리, ImpressionTracker + ImpressionTrackUnit 아키텍처

Impression Strategy Pattern RecyclerView Compose KMA-4188

배경

PR #4100에서 AmplitudeEventConsumer를 도입해 이벤트 전송 경로를 일원화했지만, 그 아래에서 Impression 수집 자체의 구조적 문제 3가지가 여전히 남아있었다.

S
스크롤 정책 혼재
Idle 감지와 Velocity 감지가 각 화면마다 다르게 구현. 동일한 목적의 ScrollListener가 4개 이상 존재.
V
노출 판정 산발
70% 비율, 중첩 뷰 교차, 단순 visible 등 판정 로직이 ViewHolder마다 하드코딩.
D
View/Compose 이중 구조
Android View 기반과 Compose 기반의 Impression 로직이 완전히 분리되어 코드 중복.

결과적으로 새 화면을 추가할 때마다 "어떤 ScrollListener를 쓰지?", "노출 판정은 어떻게 하지?"를 매번 결정해야 했고, 그 결정이 ViewHolder 곳곳에 산발적으로 흩어졌다. 146개 파일, +2357/-1978 라인의 PR로 이 구조를 전면 재설계했다.

새 아키텍처

핵심 설계 원칙: 정책(언제, 어떻게)과 대상(무엇을)을 분리한다.

요구사항 설계
스크롤 감지 방식이 화면마다 다름 ScrollImpressionStrategy (Idle / Velocity)
뷰 노출 판정 기준이 다양함 ViewImpressionStrategy (Always / Ratio / Nested)
View와 Compose를 동일하게 추적 ImpressionTrackUnit (AndroidView / Compose)
ViewHolder마다 추적 대상이 다름 ImpressionCapturable 인터페이스
이벤트 중복 전송 방지 AnalyticsEventConsumer

각 행이 독립적인 sealed interface 또는 interface로 구현되며, 조합 방식으로 동작한다. 하나의 RecyclerView에서 스크롤 정책과 노출 정책을 자유롭게 선택할 수 있다.

Strategy 패턴 -- ScrollImpressionStrategy

"스크롤이 어떤 조건일 때 Impression을 수집할 것인가"를 정의한다.

Idle
스크롤 정지(SCROLL_STATE_IDLE) 시 화면에 보이는 항목을 수집. 가장 단순하고 확실한 방식.
Velocity
스크롤 속도를 계산해서 기준 이하로 느려지면 수집. 빠르게 스와이프하는 항목은 무시.
sealed interface ScrollImpressionStrategy {
    /** 스크롤이 완전히 멈출 때 수집 */
    data object Idle : ScrollImpressionStrategy

    /** 스크롤 속도가 기준 이하일 때 수집 */
    data class Velocity(
        val thresholdPixelsPerSecond: Float,
    ) : ScrollImpressionStrategy
}

각 Strategy에 대응하는 ScrollListener가 1:1로 매핑된다.

Idle Listener

IdleImpressionScrollListener -- SCROLL_STATE_IDLE일 때 화면의 ViewHolder를 순회하며 captureImpressionUnits() 호출

Velocity Listener

VelocityImpressionScrollListener -- ScrollVelocityDetector로 속도 측정, 임계값 이하면 수집 트리거

IdleImpressionScrollListener 핵심 로직

class IdleImpressionScrollListener(
    private val eventConsumer: AnalyticsEventConsumer,
) : RecyclerView.OnScrollListener() {

    override fun onScrollStateChanged(
        recyclerView: RecyclerView,
        newState: Int,
    ) {
        if (newState != RecyclerView.SCROLL_STATE_IDLE) return

        val layoutManager = recyclerView.layoutManager as? LinearLayoutManager
            ?: return
        val first = layoutManager.findFirstVisibleItemPosition()
        val last = layoutManager.findLastVisibleItemPosition()

        (first..last).mapNotNull { position ->
            recyclerView.findViewHolderForAdapterPosition(position)
        }.filterIsInstance<ImpressionCapturable>()
         .flatMap { it.captureImpressionUnits() }
         .mapAnalyticsEventsIfVisible()
         .forEach { eventConsumer.send(it) }
    }
}

Strategy 패턴 -- ViewImpressionStrategy

"뷰가 어떤 상태일 때 노출된 것으로 판정할 것인가"를 정의한다.

Always
getVisibleHeight > 0이면 무조건 노출. 단순 배너나 전체 노출 항목에 적합.
Ratio(percentage)
뷰 높이의 지정 비율(e.g. 70%) 이상이 화면에 보여야 노출 판정. 상품 카드 등에 사용.
Nested(nestedView)
중첩 뷰(e.g. 가로 스크롤 내부)와 타겟 뷰 간 교차 영역을 계산해 판정.
sealed interface ViewImpressionStrategy {

    fun isVisible(targetView: View): Boolean

    /** 화면에 1px이라도 보이면 true */
    data object Always : ViewImpressionStrategy {
        override fun isVisible(targetView: View): Boolean =
            targetView.getVisibleHeight() > 0
    }

    /** 지정 비율 이상 노출 시 true */
    data class Ratio(
        val percentage: Int,
    ) : ViewImpressionStrategy {
        override fun isVisible(targetView: View): Boolean {
            val visibleHeight = targetView.getVisibleHeight()
            val totalHeight = targetView.height
            if (totalHeight <= 0) return false
            return (visibleHeight * 100 / totalHeight) >= percentage
        }
    }

    /** 중첩 뷰와의 교차 영역으로 판정 */
    data class Nested(
        val nestedView: View,
    ) : ViewImpressionStrategy {
        override fun isVisible(targetView: View): Boolean {
            val targetRect = Rect()
            val nestedRect = Rect()
            targetView.getGlobalVisibleRect(targetRect)
            nestedView.getGlobalVisibleRect(nestedRect)
            return targetRect.intersect(nestedRect)
        }
    }
}

ImpressionTrackUnit + ImpressionCapturable

두 Strategy가 "언제, 어떻게"를 결정한다면, ImpressionTrackUnit은 "무엇을"을 정의한다.

ImpressionTrackUnit

sealed interface ImpressionTrackUnit {
    fun isVisible(): Boolean
    fun getAnalyticsEvents(): List<AnalyticsEvent>
}

/** Android View 기반 — targetView + strategy + eventProvider */
data class AndroidViewImpressionTrackUnit(
    val targetView: View,
    val viewImpressionStrategy: ViewImpressionStrategy,
    val eventProvider: () -> List<AnalyticsEvent>,
) : ImpressionTrackUnit {
    override fun isVisible(): Boolean =
        viewImpressionStrategy.isVisible(targetView)
    override fun getAnalyticsEvents(): List<AnalyticsEvent> =
        eventProvider()
}

/** Compose 기반 — Compose가 자체 visibility 관리 */
data class ComposeImpressionTrackUnit(
    val eventProvider: () -> List<AnalyticsEvent>,
) : ImpressionTrackUnit {
    override fun isVisible(): Boolean = true
    override fun getAnalyticsEvents(): List<AnalyticsEvent> =
        eventProvider()
}

AndroidViewImpressionTrackUnit은 targetView와 ViewImpressionStrategy를 조합해서 노출 판정을 위임한다. ComposeImpressionTrackUnit은 Compose가 자체적으로 visibility를 관리하므로 isVisible()이 항상 true를 반환한다.

ImpressionCapturable

ViewHolder가 구현하는 인터페이스. "이 ViewHolder에서 어떤 이벤트를 어떤 기준으로 추적할지"를 정의한다.

interface ImpressionCapturable {
    /** 이 ViewHolder가 추적하는 ImpressionTrackUnit 목록 */
    fun captureImpressionUnits(): List<ImpressionTrackUnit>

    /** 편의 확장 — 노출된 unit들의 이벤트를 flat하게 반환 */
    fun getAnalyticsEvents(): List<AnalyticsEvent> =
        captureImpressionUnits()
            .flatMap { it.getAnalyticsEvents() }
}
ViewHolder 구현 예시

하나의 ViewHolder가 여러 TrackUnit을 반환할 수 있다. 예를 들어 상품 카드 ViewHolder는 카드 자체의 Impression + 내부 배지의 Impression을 각각 다른 Strategy로 추적할 수 있다.

class ProductCardViewHolder(
    private val binding: ItemProductCardBinding,
) : RecyclerView.ViewHolder(binding.root),
    ImpressionCapturable {

    private var product: Product? = null

    override fun captureImpressionUnits(): List<ImpressionTrackUnit> {
        val product = product ?: return emptyList()
        return listOf(
            AndroidViewImpressionTrackUnit(
                targetView = binding.root,
                viewImpressionStrategy = ViewImpressionStrategy.Ratio(
                    percentage = 70,
                ),
                eventProvider = {
                    listOf(product.toImpressionEvent())
                },
            ),
        )
    }
}

전체 흐름

  • 1 RecyclerView에 ScrollImpressionStrategy를 설정한다. 화면 특성에 따라 Idle 또는 Velocity를 선택.
  • 2 대응하는 ScrollListener(IdleImpressionScrollListener 또는 VelocityImpressionScrollListener)가 트리거 조건을 감지한다.
  • 3 화면에 보이는 ViewHolder들을 순회하며 ImpressionCapturable인 것들에서 captureImpressionUnits()를 호출한다.
  • 4 각 ImpressionTrackUnit의 isVisible()로 노출 판정. ViewImpressionStrategy에 위임.
  • 5 AnalyticsEventConsumer가 markConsumed로 중복 체크 후, 새 이벤트만 전송한다.
ImpressionTracker는 RecyclerView의 tag에 저장되어 lifecycle이 RecyclerView와 동일하게 관리된다. 화면이 사라지면 자연스럽게 GC 대상.

제거의 핵심
  • Amplitude 특화 Consumer 3종 -- SDK 이름이 인터페이스에 박혀있던 결합도를 AnalyticsEventConsumer로 일반화
  • 화면별 ScrollListener -- 각 화면이 자체 구현하던 것을 2가지 Strategy ScrollListener로 통합
  • ViewHolder 확장 함수 -- 산발적 hasBeenHandledAmplitudeEvent 체크를 ImpressionCapturable 계약으로 대체

한계

  • ViewHolder boilerplate -- Impression 추적이 필요한 모든 ViewHolder가 ImpressionCapturable을 구현해야 한다. captureImpressionUnits() 구현이 반복적일 수 있으며, ViewHolder 수가 많은 화면에서는 작업량이 체감된다.
  • Velocity 임계값 하드코딩 -- VelocityImpressionScrollListener의 속도 임계값이 코드에 고정되어 있다. 화면별로 최적의 임계값이 다를 수 있으나 현재는 설정을 변경할 수 없는 구조.
  • Compose 통합의 한계 -- ComposeImpressionTrackUnit은 isVisible()이 항상 true를 반환한다. Compose 자체의 visibility 판정을 Strategy로 끌어오지 못했으며, RecyclerView 내부에 Compose가 중첩된 하이브리드 구조에서만 의미를 가진다.

정리

Impression 이벤트의 수집 과정을 "언제(ScrollImpressionStrategy)", "어떻게(ViewImpressionStrategy)", "무엇을(ImpressionTrackUnit)"로 분리하고, 각 ViewHolder가 ImpressionCapturable을 통해 자신의 추적 정책을 선언하는 구조.
