화면 하나, Delegator 셋

38개 화면, 42개 Delegator 인터페이스, 170개 파일 변경. 하나의 네비게이션 시스템을 재설계하는 데 필요했던 숫자다.

컬리 앱에서 상품 상세 화면으로 진입하는 경로는 여러 개다. 홈 탭에서 상품을 탭하면 HomeDelegator가, 배너를 탭하면 BannerDelegator가, 딥링크(deep link)로 진입하면 ProductNavDelegator가 각각 Intent를 생성한다. 같은 화면으로 가는 로직이 세 곳에 흩어져 있었다.

새 화면을 추가하려면 최소 세 파일을 세 모듈에 걸쳐 작성해야 했다. :features 모듈에 Delegator 인터페이스를 정의하고, :app 모듈에 구현체를 작성하고, Hilt 모듈에 바인딩을 등록한다. 딥링크가 필요하면 :navigation 모듈에 NavDelegator까지 추가되어 파일이 다섯 개로 늘어난다.

이 구조를 Destination 기반 단일 네비게이션 시스템으로 재설계했다. Delegator 42개를 제거하고, sealed interface 하나와 KurlyNavigator 인터페이스 하나로 대체했다. 이 글에서는 왜 이 설계를 선택했는지, 어떻게 구축하고 마이그레이션했는지, 그리고 무엇을 얻고 무엇을 잃었는지를 다룬다.

Delegator 패턴 해부

기존 네비게이션 구조는 Delegator 패턴과 NavDelegator 패턴이 혼재했다. 4년간 화면이 추가될 때마다 인터페이스와 구현체가 한 쌍씩 늘어났고, 최종적으로 27개 Delegator 인터페이스와 5개 NavDelegator, 그리고 이를 조합하는 Navigator 객체가 존재했다.

Delegator 패턴의 기본 구조는 Feature 모듈의 인터페이스와 App 모듈의 구현체로 이루어진다.

// :features — 인터페이스 정의
interface HomeDelegator {
    fun navigateToHome(context: Context, site: String?)
}

// :app — 구현체
class HomeDelegatorImpl @Inject constructor(
    private val homeIntentGenerator: HomeIntentGenerator,
) : HomeDelegator {
    override fun navigateToHome(context: Context, site: String?) {
        val intent = homeIntentGenerator.generateIntent(context, site)
        context.startActivity(intent)
    }
}

호출부는 두 가지 패턴이 공존했다. NavDelegator를 경유하는 방식과 Delegator를 직접 호출하는 방식이다.

// NavDelegator 경유 — Navigator 객체의 sub-delegator 조합
navigator.home.startMainActivity(context)
navigator.cart.startCartActivity(context)
navigator.product.startProductDetailActivity(context, productNo)

// Delegator 직접 호출
bannerDelegator.handleUrl(context, url)
homeDelegator.navigateToCategory(context, categoryId)
dialogDelegator.alert(fragmentManager, title, buttonText)

이 구조의 문제는 세 가지였다.

첫째, 진입 로직의 파편화. 상품 상세 화면 하나를 호출하는 코드가 ProductNavDelegator, BannerDelegator, HomeDelegator, CategoryDelegator에 각각 존재했다. 파라미터 전달 방식이 Delegator마다 달랐고, 하나를 수정하면 나머지를 찾아 동기화해야 했다.

둘째, 새 화면 추가 비용. Interface 1개 + Impl 1개 + Hilt 바인딩 1건이 기본이었다. 딥링크가 필요하면 NavDelegator Interface + Impl이 추가되어 파일이 다섯 개까지 늘어났다.

셋째, 딥링크 일관성 부재. 딥링크 파싱 로직이 각 NavDelegator에 산재했다. kurly://product?no=123을 처리하는 코드와 kurly://home?site=market을 처리하는 코드가 서로 다른 패턴을 사용했다.

Compose Navigation 대신 sealed interface를 선택한 이유

네비게이션 시스템을 재설계할 때 가장 먼저 검토한 것은 Compose Navigation이었다. Jetpack의 공식 솔루션이고, 타입 안전한 route 시스템을 제공한다.

그러나 현실적 제약이 있었다. 컬리 앱은 Activity 기반 아키텍처다. 38개 화면 대부분이 독립 Activity로 구현되어 있다. Compose Navigation을 전면 도입하려면 단일 Activity 구조로의 전환이 선행되어야 하고, 이는 현재 앱 규모에서 별도의 대형 마이그레이션이 된다.

단계적 전환 전략을 세웠다.

Phase 1 (현재): Delegator 패턴 → Destination 기반 KurlyNavigator
Phase 2 (미래): Destination → Compose Navigation routes로 확장

Destination sealed interface를 중간 단계로 선택한 이유는 세 가지다.

컴파일 타임 안전성. sealed interface의 when 표현식은 모든 화면 유형을 처리했는지 컴파일러가 검증한다. Delegator에서는 런타임에서야 발견되던 누락을 빌드 시점에 잡을 수 있다.

파라미터 타입 안전성. Destination.Screen.ProductDetail(productNo: Long)은 파라미터 이름과 타입을 강제한다. startProductDetailActivity(context, "123")처럼 String으로 전달하던 패턴에서 Long 타입으로 변환하여 컴파일 타임에 검증한다.

딥링크 양방향 변환. DeepLinkSpec<T>를 Destination의 companion object에 구현하면, 딥링크 파싱과 생성이 한 곳에서 관리된다. 파싱과 생성이 항상 대칭이므로, 딥링크 URL을 만들었다가 다시 파싱하면 원래 Destination으로 복원되는 라운드트립이 보장된다.

새 시스템 구축

새 네비게이션 시스템은 네 개의 핵심 컴포넌트로 구성된다.

네비게이션 처리 흐름

Screen 전환  Destination.Screen → KurlyNavigator → ScreenResolver → IntentGenerator → startActivity

Overlay 표시  Destination.Overlay → KurlyNavigator → OverlayResolver → Fragment.show()

URL 전환  URL → DestinationRegistry → DeepLinkSpec.fromUri() → Destination → Screen 전환 경로와 합류

Destination: 화면의 타입 안전한 표현

모든 화면을 sealed interface 계층으로 정의했다. Screen은 Activity로 전환되는 화면, Overlay는 Dialog/BottomSheet로 표시되는 화면이다.

sealed interface Destination {

    sealed interface Screen : Destination {
        data class ProductDetail(val productNo: Long) : Screen
        data object Cart : Screen
        data class Home(val site: String? = null) : Screen
        // ... 38개 Screen
    }

    sealed interface Overlay : Destination {
        data class Alert(val title: String, val buttonText: String) : Overlay
        // ... 3개 Overlay
    }
}

파라미터가 있는 화면은 data class, 없는 화면은 data object로 구분했다. 이 구분 덕분에 Cart처럼 파라미터가 없는 화면은 싱글톤으로 사용하고, ProductDetail(productNo = 123)처럼 파라미터가 있는 화면은 인스턴스를 생성한다.

DeepLinkSpec: 양방향 딥링크 변환

딥링크를 지원하는 화면은 companion object에서 DeepLinkSpec<T>를 구현한다. fromUri와 toUri가 한 곳에 있으므로 파싱과 생성이 항상 대칭이다.

data class ProductDetail(val productNo: Long) : Screen {
    companion object : DeepLinkSpec<ProductDetail> {
        override val host = "product"
        override fun fromUri(uri: Uri): ProductDetail? {
            val no = uri.getQueryParameter("no")?.toLongOrNull()
                ?: return null
            return ProductDetail(no)
        }
        override fun toUri(destination: ProductDetail): Uri =
            buildDeepLink {
                host("product")
                appendQuery("no", destination.productNo.toString())
            }
    }
}

ProductDetail.toUri(dest)로 생성한 URI를 다시 ProductDetail.fromUri(uri)로 파싱하면 원래 Destination이 복원된다. 이 라운드트립은 테스트로 검증했다.[1]

KurlyNavigator: 단일 진입점

42개 Delegator 인터페이스를 5개 메서드로 통합했다.

interface KurlyNavigator {
    fun navigate(context: Context, destination: Destination.Screen)
    fun resolve(context: Context, destination: Destination.Screen): Intent
    fun navigate(context: Context, destination: Destination.Screen,
                 launcher: ActivityResultLauncher<Intent>)
    fun show(fragmentManager: FragmentManager, destination: Destination.Overlay)
    fun navigate(context: Context, url: String,
                 fallback: FallbackAction = FallbackAction.WEB_VIEW)
}

Screen 전환, 결과 대기 전환, Overlay 표시, URL 기반 전환 — 네 가지 유스케이스를 하나의 인터페이스에서 처리한다. 호출부에서는 어떤 유형의 네비게이션이든 navigator 하나만 주입받으면 된다.

Resolver: Intent 생성 전략

KurlyNavigatorImpl은 Destination 타입에 따라 적합한 Resolver를 찾아 Intent를 생성한다. 핵심은 기존 IntentGenerator를 재활용하는 것이다.

class ProductDetailResolver @Inject constructor(
    private val intentGenerator: ProductIntentGenerator,
) : ScreenResolver<Destination.Screen.ProductDetail> {
    override fun resolve(
        context: Context,
        destination: Destination.Screen.ProductDetail,
    ): Intent {
        val uri = Destination.Screen.ProductDetail.toUri(destination)
        return intentGenerator.generateIntent(context, uri)
    }
}

Resolver는 Destination의 toUri로 파라미터를 URI로 변환한 뒤, 기존 IntentGenerator에 위임한다. Delegator가 하던 일(파라미터 조합 → Intent 생성)을 Resolver가 대신하되, Intent 생성 로직 자체는 건드리지 않았다.

Hilt의 @IntoMap 멀티바인딩으로 Resolver를 등록하면, KurlyNavigatorImpl이 Destination의 KClass를 키로 적합한 Resolver를 찾는다.

class KurlyNavigatorImpl @Inject constructor(
    private val screenResolvers: Map<KClass<out Destination.Screen>,
        @JvmSuppressWildcards ScreenResolver<out Destination.Screen>>,
    private val uriNavigator: UriNavigator,
) : KurlyNavigator {

    override fun resolve(context: Context, destination: Destination.Screen): Intent {
        val resolver = screenResolvers[destination::class]
            ?: error("No ScreenResolver for ${destination::class.simpleName}")
        return (resolver as ScreenResolver<Destination.Screen>)
            .resolve(context, destination)
    }

    override fun navigate(context: Context, url: String, fallback: FallbackAction) {
        val destination = DestinationRegistry.fromUri(url.toUri())
        if (destination is Destination.Screen) navigate(context, destination)
        else uriNavigator.navigate(context, url, fallbackAction = fallback)
    }
}

URL 기반 네비게이션에서는 DestinationRegistry가 URI의 host를 키로 등록된 DeepLinkSpec 목록에서 매칭을 시도한다. 매칭되면 타입 안전한 Destination 경로를 타고, 매칭되지 않으면 기존 UriNavigator로 폴백한다. 새 시스템과 기존 시스템이 공존할 수 있는 구조다.

42개 Delegator 제거

새 시스템을 구축한 후, 기존 Delegator를 제거하는 단계가 남았다. 102개 파일에서 42개 인터페이스와 구현체를 삭제하고, 모든 호출부를 KurlyNavigator로 전환했다.

3단계 커밋 전략을 사용했다. 순서에는 의도가 있다.

마이그레이션 커밋 전략

1단계: 의존 루트 차단 — :navigation 모듈의 5개 NavDelegator와 Navigator 객체를 제거한다. 다른 모듈이 이 인터페이스에 의존하지 않도록 먼저 루트를 끊는다.

2단계: 저사용 Delegator — :features 모듈의 저사용 Delegator(InquiryComposeDelegator, PickListDelegator 등)를 전환한다. 호출 지점이 1-2곳이므로 리스크가 낮다.

3단계: 고사용 Delegator — Banner, Home, Brand, Category, OrderResult처럼 호출 지점이 많은 Delegator를 전환한다. 변경 범위가 넓어 별도 커밋으로 격리했다.

호출부 변환은 일관된 패턴을 따른다.

Before
navigator.home.startMainActivity(ctx)
navigator.cart.startCartActivity(ctx)
navigator.product.startProductDetailActivity(ctx, no)
bannerDelegator.handleUrl(ctx, url)
dialogDelegator.alert(fm, title, btn)
After
navigator.navigate(ctx, Destination.Screen.Home())
navigator.navigate(ctx, Destination.Screen.Cart)
navigator.navigate(ctx, Destination.Screen.ProductDetail(no))
navigator.navigate(ctx, url)
navigator.show(fm, Destination.Overlay.Alert(title, btn))

Before의 호출부는 어떤 Delegator의 어떤 메서드를 호출해야 하는지를 알아야 한다. After에서는 navigator.navigate 또는 navigator.show 두 가지 패턴으로 수렴한다. 호출부만 보면 어떤 화면으로 가는지 즉시 파악된다.

결과: 1,616줄 삭제, 395줄 추가. 순 1,221줄 감소.[2]

퇴보. Resolver 파일이 36개 추가되었다. 각 화면마다 Resolver 클래스가 필요하므로 파일 수 자체는 늘어났다. 다만 Resolver는 평균 10-15줄의 단순한 위임 코드로, Delegator의 Interface + Impl 조합(평균 40-60줄)보다 구조가 단순하다.

Destination.kt 파일이 511줄이라는 점도 주의할 부분이다. 38개 Screen과 3개 Overlay, 그리고 딥링크를 지원하는 화면들의 DeepLinkSpec 구현이 한 파일에 모여 있다. 화면 수가 계속 늘어나면 Destination.Screen.kt와 Destination.Overlay.kt로 분리하거나, 패키지 단위로 나누는 것을 검토해야 한다.

다음 목적지

이 설계의 목적은 Delegator 제거 자체가 아니라, Compose Navigation으로의 점진적 전환 경로를 확보하는 것이다.

Destination sealed interface는 Compose Navigation의 route와 구조적으로 유사하다. Destination.Screen.ProductDetail(productNo = 123)은 Compose Navigation 2.8+의 @Serializable data class ProductDetail(val productNo: Long) route와 거의 동일한 형태다. Activity 기반 화면을 하나씩 Composable로 전환하면서 Resolver를 NavGraphBuilder로 교체하면 된다. 호출부의 navigator.navigate(context, Destination.Screen.XXX)는 변경할 필요가 없다.

핵심 정리
  • Delegator 42개 → KurlyNavigator 1개: 파편화된 네비게이션 로직을 단일 인터페이스로 통합
  • DeepLinkSpec 양방향 변환: 딥링크 파싱과 생성이 한 곳에서 관리되어 일관성 보장
  • 순 1,221줄 삭제: 새 시스템 구축(+2,022줄) 후 기존 시스템 제거(-1,616줄)
  • Compose Navigation 전환 발판: Activity 기반에서 즉시 가능한 중간 단계 설계

이 패턴이 적합한 경우: Activity 기반 아키텍처에서 Compose Navigation 전면 전환이 즉시 불가능하고, Delegator/Navigator 패턴이 10개 이상 누적된 앱.

적합하지 않은 경우: 이미 단일 Activity + Compose Navigation으로 전환 완료된 앱, 화면 수가 10개 미만인 소규모 앱(Resolver 오버헤드가 본체보다 클 수 있다), 또는 WebView 기반 하이브리드 앱(네이티브 네비게이션 레이어가 얇으면 이 구조가 과도하다).
