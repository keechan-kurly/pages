startActivity()가 조용히 차단될 때 — PIP 모드와 백그라운드 액티비티 시작 제한

Samsung 보안 패치 이후 라이브커머스 상품 상세 전환이 실패한 원인을 추적하고, PIP 모드에서의 액티비티 시작 타이밍 함정을 분석한다.

PIP 모드에서 startActivity()를 호출하면 새 화면이 열린다고 생각한다. 하지만 Samsung One UI 보안 패치 이후, 이 호출은 아무 예외도 던지지 않은 채 조용히 무시되었다.

증상

Samsung One UI 보안 패치(F731NKSS5EYGB1) 이후, 특정 Samsung 기기에서 라이브커머스 상품 선택 시 상세 화면으로 전환되지 않는 이슈가 보고되었다. 사용자가 라이브 방송 중 상품을 탭해도 아무 반응이 없었다.

Logcat에서 발견한 단서는 한 줄이었다.

ActivityTaskManager: Background activity launch blocked!

"백그라운드 액티비티 시작이 차단되었다" — 포그라운드에서 동작 중인 라이브커머스 화면에서 이 로그가 나온다는 것은 직관에 어긋났다. 사용자가 보고 있는 화면에서 버튼을 눌렀는데, 시스템이 이를 "백그라운드"로 판정했다는 의미였다.

배경: 백그라운드 액티비티 시작 제한

Android 10(API 29)부터 백그라운드 액티비티 시작 제한(Background Activity Start Restrictions2)이 도입되었다. 악성 앱이 백그라운드에서 피싱 화면이나 광고를 띄우는 것을 방지하기 위한 보안 정책이다.

일반적으로 포그라운드 앱은 이 제한의 영향을 받지 않는다. 사용자가 보고 있는 화면에서 startActivity()를 호출하면 정상 동작한다.

그런데 "포그라운드"의 정의에 함정이 있다. PIP(Picture-in-Picture) 모드에 진입한 액티비티는 화면에 여전히 보이지만, 시스템은 이를 일시 정지(Paused) 상태로 취급한다3. 일시 정지된 액티비티는 포그라운드가 아니다.

Resumed (포그라운드) → enterPipMode() → Paused (PIP)
↑ 화면에 보이지만 시스템은 "백그라운드"로 취급

Samsung의 보안 패치는 이 정책을 더 엄격하게 적용한 것으로 추측된다. 기존에는 PIP 상태에서도 startActivity()가 허용되었지만, 패치 이후에는 차단되었다.

근본 원인: PIP 전환의 타이밍 함정

라이브커머스에서 상품을 선택하면 라이브 화면은 PIP로 전환하고, 상품 상세 화면을 새로 열어야 한다. 기존 코드는 이 과정에서 실행 순서를 의도적으로 지연했다.

// 기존 코드: Intent를 저장하고 PIP 전환 후 실행
private var nextIntent: Intent? = null

private fun startActivityWithChangePipMode(intent: Intent) {
    if (isSupportPipMode() && hasPipPermission()) {
        nextIntent = intent.flagNewTask()  // 1. Intent 저장
        enterPipMode()                      // 2. PIP 진입
    } else {
        startActivity(intent)
    }
}

PIP 진입과 동시에 startActivity()를 호출하면 PIP 화면 안에서 화면 전환이 일어나는 시각적 문제가 있었다. 이를 피하기 위해 Intent를 저장해두고, PIP 전환이 완료된 시점에 실행하도록 설계한 것이다.

// PIP 전환 완료 콜백에서 저장된 Intent 실행
override fun onPictureInPictureModeChanged(isInPipMode: Boolean) {
    if (isInPipMode) {
        viewModel.play()
        hideActionBar()
        closeAllDialogFragments()

        if (nextIntent != null) {
            startActivity(nextIntent)  // 3. 여기서 실행
            nextIntent = null
        }
    } else {
        showActionBar()
    }
}

문제는 3번 시점이다. onPictureInPictureModeChanged(true) 콜백이 호출되는 순간, 액티비티는 이미 PIP 모드 — 즉 Paused 상태다. Samsung 보안 패치는 이 상태에서의 startActivity() 호출을 백그라운드 액티비티 시작으로 판정하여 차단했다.

1. startActivityWithChangePipMode() 호출
↓
2. nextIntent에 Intent 저장
↓
3. enterPipMode() → 액티비티 상태: Resumed → Paused
↓
4. onPictureInPictureModeChanged(true) 콜백
↓
5. startActivity(nextIntent) ← 여기서 차단!
액티비티가 Paused 상태 → "백그라운드"로 판정

수정: 호출 순서 역전

수정은 단순하다. startActivity()를 enterPipMode() 앞으로 이동시켰다.

Before
nextIntent = intent.flagNewTask()
enterPipMode()
// → onPipModeChanged에서 실행
After
startActivity(intent.flagNewTask())
enterPipMode()

startActivity()가 호출되는 시점에 라이브커머스 액티비티는 아직 Resumed 상태다. 포그라운드에서의 호출이므로 백그라운드 제한에 걸리지 않는다.

nextIntent 변수와 onPictureInPictureModeChanged의 지연 실행 로직은 더 이상 필요하지 않으므로 제거했다. 변경량은 +2줄, -11줄이다.

이 수정으로 기존에 우려했던 "PIP 전환과 화면 전환이 동시에 일어나는" 시각적 문제가 재발할 수 있다. 하지만 실제 기기에서 테스트한 결과, FLAG_NEW_TASK로 시작된 액티비티는 별도 태스크에서 열리므로 PIP 전환과 시각적으로 충돌하지 않았다.

교훈

1. PIP 모드 진입 = 백그라운드 전환이다

화면에 보인다고 포그라운드가 아니다. PIP 액티비티는 Paused 상태이며, 시스템은 이를 백그라운드로 취급할 수 있다. PIP 모드에서 다른 액티비티를 시작해야 한다면, PIP 진입 전에 호출을 완료해야 한다.

2. OEM 보안 패치는 기존 동작을 깨뜨릴 수 있다

Android 10에서 도입된 백그라운드 액티비티 시작 제한은 제조사별로 적용 수준이 달랐다. Samsung은 보안 패치를 통해 이 정책을 더 엄격하게 적용했고, 기존에 동작하던 코드가 특정 패치 이후 갑자기 실패했다. "현재 동작한다"가 "앞으로도 동작할 것이다"를 보장하지 않는다.

3. startActivity() 차단은 조용하다

startActivity()가 백그라운드 제한에 의해 차단되면 예외를 던지지 않는다. Logcat에 ActivityTaskManager: Background activity launch blocked! 로그만 남긴다. 크래시 리포트에도 잡히지 않으므로, 이 로그를 모니터링하지 않으면 사용자가 "버튼이 안 눌린다"고 보고할 때까지 문제를 인지할 수 없다.

핵심 요약
  • PIP 모드 = Paused 상태 → 시스템은 "백그라운드"로 취급한다
  • startActivity()는 PIP 진입 전에 호출해야 백그라운드 제한을 회피한다
  • 차단 시 예외 없음 → ActivityTaskManager 로그 모니터링이 유일한 탐지 수단이다
  • OEM 보안 패치를 주시하라 → 기존 동작이 깨질 수 있다
