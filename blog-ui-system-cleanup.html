같은 코드를 다섯 번 복사하고 있었다

새 피처 모듈에 에러 화면을 추가하려고 기존 코드를 참고했다. MyKurly에 51줄짜리 MyKurlyErrorScreen이 있었고, Brand에도 50줄짜리 BrandErrorScreen이 있었다. Pick에는 57줄, Alternative에도 50줄. 네 개 파일을 열어봤더니 거의 동일한 코드였다. KurlyEmptyState를 Box로 감싸고 배경색을 설정하고 정렬을 맞추는 패턴이 반복되고 있었다.

에러 화면만이 아니었다. 상품 이미지 컴포넌트가 compose-core, search, widget 세 모듈에 각각 다른 이름으로 존재했다. noRippleClickable이라는 유틸 함수는 MyKurly 패키지에 갇혀 있어서 다른 피처에서는 접근조차 할 수 없었다.

피처가 늘어날수록 중복이 선형으로 증가하는 구조였다.

이 글은 43개 파일을 변경하고 818줄을 순삭제하면서, 산재된 UI 코드를 :features:core:ui 공유 레이어로 통합한 과정을 정리한다. Compose 기반 멀티모듈 프로젝트에서 비슷한 중복을 겪고 있다면 참고가 될 것이다.

Before: 각자도생의 피처 모듈

정비 전 상태를 세 가지 영역으로 나눌 수 있다.

1. 상태 화면: 피처마다 50줄씩 복사

에러·빈 화면·로딩 화면이 각 피처 모듈에 독립적으로 존재했다. 코드 구조는 거의 같지만 미세한 차이가 있었다.

MyKurlyErrorScreen.kt

BiasAlignment(0f, -0.22f) 사용
배경색: Background1
51줄

PickListErrorScreen.kt

Alignment.Center 사용
배경색: 없음
57줄

같은 역할의 화면인데 정렬 방식이 다르고, 배경색 적용 여부도 달랐다. 디자인 시스템이 업데이트되면 다섯 곳을 찾아 다니며 하나씩 수정해야 했다.

2. 상품 컴포넌트: 세 모듈에 산재

상품 이미지, 스티커, 장바구니 숏컷 버튼이 서로 다른 모듈에 각각 구현되어 있었다.

fun VerticalProductImage(...) { ... }

// compose-core/product/Sticker.kt — 168줄

VerticalProductItemInfo는 파라미터가 13개였다. 상품명, 가격, 할인율, 리뷰, 랭킹, 숏컷 버튼까지 하나의 Composable이 모두 처리하고 있었다. search 모듈의 ProductImage는 compose-core의 VerticalProductImage와 역할은 같지만 파라미터 설계가 달랐다.

3. 피처 전용 유틸: 접근 불가

MyKurly에 정의된 noRippleClickable 확장 함수를 다른 피처에서 쓰려면 모듈 의존성을 추가하거나 복사하는 수밖에 없었다. 실제로 이름이 모호한 ThumbnailImage라는 Composable이 widget 모듈에 이미 존재했지만, 이름만으로는 상품 이미지 컴포넌트라는 사실을 알기 어려웠다.

전략: 3-Phase 정비

:features:core:ui 모듈은 이미 두 개의 하위 모듈로 구성되어 있었다.

:features:core:ui 모듈 구조

template — 화면 단위 레이아웃 (ErrorScreen, LoadingScreen 등)
widget — 재사용 가능한 개별 컴포넌트 (ProductImage, ShortCutButton 등)

template과 widget이라는 두 레이어의 역할 구분이 명확했기 때문에, 정비 범위를 이 구조에 맞춰 세 단계로 나눴다.

  • Phase 1 (KMA-7414): 피처별 상태 화면을 template 레이어에 공통화
  • Phase 2 (KMA-7415): 산재된 상품 컴포넌트를 widget 레이어로 병합
  • Phase 3 (KMA-7416): 호출부 마이그레이션 완료 + 구 코드 삭제

Phase 1과 2는 새 컴포넌트를 만들되 기존 코드를 건드리지 않는 방식으로 진행했다. 기존 코드와 새 코드가 공존하는 구간을 의도적으로 두어, 중간에 빌드가 깨지지 않도록 했다. Phase 3에서 호출부를 일괄 교체한 뒤 구 코드를 삭제했다.

Phase 1 — 상태 화면 통합

에러 화면, 빈 화면, 로딩 화면 세 가지 공통 템플릿을 만들었다. 핵심 설계 원칙은 컴포넌트 슬롯이다. 피처별로 미세하게 다른 부분(분석 이벤트, 추가 UI 등)은 content 람다로 주입받는다.

@Composable
fun ErrorScreen(
    modifier: Modifier = Modifier,
    onRetryClick: () -> Unit,
) {
    Box(
        modifier = modifier
            .background(BackgroundColorTokens.Background1.toColor()),
        contentAlignment = Alignment.Center,
    ) {
        KurlyEmptyState(
            title = stringResource(R.string.error_title),
            content = stringResource(R.string.error_content),
            buttonText = stringResource(R.string.error_retry),
            onClickButton = onRetryClick,
        )
    }
}

호출부는 이렇게 단순해졌다.

Before — MyKurlyScreen.kt
when (uiState) {
    is Error ->
        MyKurlyErrorScreen(
            modifier = Modifier.fillMaxSize(),
            onClick = { /* retry */ },
        )
After — MyKurlyScreen.kt
when (uiState) {
    is Error ->
        ErrorScreen(
            modifier = Modifier.fillMaxSize(),
            onRetryClick = { /* retry */ },
        )

표면적으로는 함수 이름만 바뀐 것처럼 보이지만, 핵심은 소스가 한 곳으로 수렴했다는 점이다. 이전에는 MyKurlyErrorScreen, BrandErrorScreen, PickListErrorScreen 각각이 정렬·배경·문구를 독자적으로 결정했다. 이제는 ErrorScreen 하나가 이 결정을 내리고, 4개 피처(MyKurly, Brand, Pick, Alternative)가 그것을 그대로 사용한다.

이 과정에서 4개 파일(210줄)을 삭제하고 3개 템플릿(126줄)으로 대체했다. LoadingScreen은 기존에도 존재했지만 fillMaxSize가 하드코딩되어 있어서, backgroundColor 파라미터를 추가하고 사이징은 호출부가 결정하도록 개선했다.

Phase 2 — 상품 컴포넌트 병합

가장 복잡한 단계였다. compose-core/product/ 패키지(642줄)와 search/.../Composable.kt(269줄)에 산재된 상품 관련 Composable을 widget 패키지로 모았다.

ProductImage: 두 구현을 하나로

compose-core의 VerticalProductImage와 search의 ProductImage는 둘 다 상품 이미지를 보여주는 역할이었지만, search 쪽이 aspectRatio와 additionalView 슬롯을 지원해서 더 유연했다. 두 구현의 요구사항을 합쳐 widget 쪽 ProductImage로 통합했다.

@Composable
fun ProductImage(
    imageUrl: String,
    modifier: Modifier = Modifier,
    stickers: List<StickerData> = emptyList(),
    isBuyPossible: Boolean = true,
    notPurchaseMessage: String? = null,
)

기존 VerticalProductImage와 비교하면 달라진 점이 두 가지다.

  • 사이즈를 modifier로 외부 주입. 기존에는 112x145dp가 함수 내부에 하드코딩되어 있었다. 이제 호출부가 Modifier.size(width = 112.dp, height = 145.dp)로 명시한다.
  • 스티커 타입을 도메인 모델에서 분리. List<Sticker>(도메인) 대신 List<StickerData>(UI 전용 모델)를 받는다. 변환은 확장 함수 한 줄로 처리한다.

StickerDataMapper: 도메인-UI 경계

fun List<Sticker>.toStickerData(): List<StickerData> =
    map { sticker ->
        StickerData(
            type = StickerType.fromDomain(sticker.type),
            content = sticker.content,
        )
    }

31줄짜리 작은 파일이지만, 이 확장 함수가 없으면 widget 모듈이 도메인 스티커 타입에 직접 의존하게 된다. UI 레이어의 StickerData가 도메인 Sticker를 몰라도 되도록 경계를 분리하는 역할이다.

13개 파라미터 → 컴포지션 분리

가장 큰 구조적 변화는 VerticalProductItemInfo(13개 파라미터)를 세 컴포넌트로 분리한 것이다.

Before — 하나의 Composable

VerticalProductItemInfo
13개 파라미터
이미지·가격·리뷰·랭킹·숏컷 모두 처리

After — 세 컴포넌트 조합

ProductInfo — 이름, 배송, 가격, 리뷰, 랭킹
ShortCutButton — 장바구니 버튼
PriceInfoData — 가격 정보 데이터 클래스

호출부에서의 변화를 보면 의도가 더 명확해진다.

// After: 역할별로 분리된 컴포넌트 조합
Column(verticalArrangement = Arrangement.spacedBy(2.dp)) {
    ProductInfo(
        name = name,
        deliveryType = null,
        priceData = PriceInfoData(
            discountedPrice = discountedPrice,
            discountRate = discountRate,
            finalPrice = getFinalPrice(),
            isMultiplePrice = isMultiplePrice,
        ),
        reviewCount = null,
        ranking = rankingNumber,
    )

    Spacer(modifier = Modifier.weight(1f))

    ShortCutButton(
        text = shortCutType.text,
        modifier = Modifier.fillMaxWidth(),
        leadingIcon = shortCutType.icon?.let {
            ImageVector.vectorResource(it)
        },
        onClick = { /* analytics + action */ },
    )
}

PriceInfoData로 가격 관련 4개 파라미터를 묶었다. 숏컷 버튼은 별도 컴포넌트로 분리되어, 버튼이 필요 없는 화면에서는 아예 호출하지 않으면 된다. 이전에는 onClickShortCut = null을 넘겨 내부 분기를 타야 했다.

이 Phase에서 compose-core/product/ 패키지 전체(642줄)와 search/.../Composable.kt(269줄)를 삭제하고, widget 패키지에 3개 파일(145줄)을 추가했다.

Phase 3 — API 표준화

Phase 1, 2에서 새 컴포넌트를 만들었으니 Phase 3에서는 호출부 14곳을 일괄 마이그레이션하고 구 코드를 삭제했다.

noRippleClickable → clickableSingleWithNoRipple

MyKurly의 Util.kt에만 존재하던 noRippleClickable을 compose-core/extension/의 공유 확장 함수 clickableSingleWithNoRipple로 교체했다.

// Before: MyKurly 전용 유틸
import com.kurly.features.mykurly.compose.util.noRippleClickable
modifier.noRippleClickable(onClick = onToolTipClick)

// After: 공유 확장 함수
import com.kurly.compose_core.extension.clickableSingleWithNoRipple
modifier.clickableSingleWithNoRipple(onClick = onToolTipClick)

이름이 길어졌지만, 역할이 더 명확해졌다. clickableSingle은 중복 클릭 방지를 포함하고, WithNoRipple은 리플 효과 제거를 명시한다. noRippleClickable이라는 이름은 "리플을 없앤다"는 사실은 알려주지만 중복 클릭 방지 여부는 암시하지 않았다.

ThumbnailImage → ProductImage 리네이밍

widget 모듈에 이미 존재하던 ThumbnailImage를 ProductImage로 이름을 변경했다. 썸네일이라는 이름은 이 컴포넌트가 상품 이미지 + 스티커 오버레이 + 판매불가 딤 처리까지 담당한다는 사실을 드러내지 못했다.

리네이밍과 함께 template/product/ 하위의 ProductCard, ProductHorizontalListItem, ProductVerticalListItem 세 곳의 import를 일괄 변경했다.

한계와 남은 과제

이번 정비로 해결하지 못한 부분도 있다.

  • compose-core 모듈의 완전 제거는 아직이다. product/ 패키지는 삭제했지만, compose-core에는 category, analytics 등 아직 다른 코드가 남아 있다. 장기적으로 이 모듈의 역할을 features:core:ui로 완전히 이관해야 한다.
  • search 모듈의 ViewHolder 코드는 점진적 전환 중이다. Composable.kt(269줄)에서 상품 관련 코드는 삭제했지만, ViewHolder에서 Compose 컴포넌트를 호출하는 구조 자체는 RecyclerView → LazyColumn 전환이 완료될 때까지 유지된다.
  • widget 컴포넌트의 테스트 커버리지가 없다. 이번 PR은 기존 동작의 위치 이동에 집중했기 때문에 Preview 함수로만 시각적 검증을 했다. 향후 스냅샷 테스트 도입을 고려하고 있다.

핵심 교훈
  • template과 widget을 구분하라. 화면 단위 레이아웃(template)과 개별 컴포넌트(widget)를 분리하면, "이 코드는 어디에 둬야 하지?"라는 질문에 답하기 쉬워진다.
  • 파라미터 폭발은 컴포지션으로 해결하라. 13개 파라미터의 단일 Composable보다, 역할별로 분리된 3개 컴포넌트를 Column으로 조합하는 것이 더 유연하고 읽기 쉽다.
  • 구 코드 삭제는 마지막에 한다. 새 컴포넌트를 먼저 만들고, 호출부를 점진적으로 교체한 뒤, 마지막 Phase에서 일괄 삭제하면 중간에 빌드가 깨지지 않는다.
