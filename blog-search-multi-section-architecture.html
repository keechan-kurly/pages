검색 지면에 Server Driven UI 적용하기: 단일 타입에서 멀티섹션으로

구매 전환율 향상을 위해 검색 API를 v3 flat response에서 v4 multi-section 구조로 전환하고, KSP 기반 자동 매핑과 어댑터 통합으로 클라이언트 아키텍처를 재설계한 과정을 다룬다.

검색 지면의 구매 전환율을 높여야 한다. 서버가 배너, 키워드 캐스팅(keyword casting), 필터, 상품을 자유롭게 조합할 수 있어야 한다. 클라이언트는 이 조합을 사전에 알 수 없다.

이 세 가지 제약이 검색 API v3→v4 마이그레이션의 출발점이었다. 이 글에서는 단일 UI 타입만 노출하던 검색 지면을 멀티섹션 구조로 전환한 과정 — 커스텀 역직렬화(deserialization), KSP 기반 매퍼 자동 발견, 어댑터 아키텍처 통합 — 을 다룬다.

왜 바꿨나

기존 검색 지면은 상품 리스트라는 단일 UI 타입만을 노출했다. 사용자가 검색어를 입력하면 상품 목록이 나오고, 필터와 정렬은 화면 상단에 고정된 헤더로 제공되었다. 이 구조는 "검색 = 상품 나열"이라는 요구사항에는 충분히 잘 동작했다.

그러나 구매 전환율을 높이려면 검색 결과에 다양한 맥락을 함께 보여줘야 했다. 검색어에 맞는 프로모션 배너, 오타 교정(keyword casting), 뷰티컬리 전용 필터 등 — 서버가 검색어와 사용자 상황에 따라 UI 구성을 동적으로 결정할 수 있어야 했다.

문제는 기존 구조에서 새 UI 타입을 추가하려면 클라이언트 릴리즈가 필수였다는 점이다. 배너 섹션을 넣으려면 ProductPagingHeaderAdapter를 수정하고, VIEW_TYPE 상수 충돌을 확인하고, Fragment에서 데이터를 바인딩하는 코드를 추가해야 했다. "새 섹션 타입 = 앱 릴리즈"라는 종속이 서버 팀의 실험 속도를 제한하고 있었다.

Server Driven UI의 핵심은 "서버가 무엇을 보여줄지 결정하고, 클라이언트는 그것을 어떻게 그릴지만 담당한다"는 역할 분리다. 검색 지면에 이 원칙을 적용하면, 서버가 새 섹션 타입을 내려보내는 것만으로 UI 실험이 가능해진다.

기존 구조 — v3 Flat Response

v3 검색 API는 상품 리스트를 flat한 배열로 내려보냈다. 응답의 핵심 구조는 List<ContentProductListItemResponse> + 별도의 meta/banner 필드였다.

클라이언트에서는 이 flat 응답을 ConcatAdapter로 조립했다. ProductPagingHeaderAdapter가 배너, 필터, 키워드 캐스팅을 담당하고, ProductPagingListAdapter가 상품 리스트를 담당했다. 두 어댑터를 ConcatAdapter로 합쳐서 하나의 RecyclerView에 표시하는 구조였다.

v3 API Response
↓
ProductPagingHeaderAdapter
(배너 + 필터 + 키워드캐스팅)
+
ProductPagingListAdapter
(상품 리스트)
↓
ConcatAdapter

이 구조의 한계는 세 가지였다.

첫째, 헤더 어댑터의 비대화. 배너, 필터, 키워드 캐스팅, 정렬 — 서로 다른 관심사를 하나의 ProductPagingHeaderAdapter가 모두 관리했다. VIEW_TYPE 상수가 26번부터 32번까지 퍼져 있었고, 새 타입 추가 시 충돌 여부를 수동으로 확인해야 했다.

둘째, 클라이언트 주도 조립. 어떤 섹션을 어떤 순서로 보여줄지를 클라이언트가 결정했다. 배너 삽입 위치, 키워드 캐스팅 표시 조건 등의 로직이 SearchProductPagingSource에 하드코딩되어 있었다.

셋째, Fragment 비대화. ProductPagingListFragment가 두 어댑터의 데이터 바인딩, StickyHeader 설정, 메타 데이터 렌더링을 모두 담당하여 코드가 복잡해지고 있었다.

설계 전략 — Server Driven Multi-Section

v4 API는 응답 구조 자체를 바꿨다. flat한 상품 배열 대신, 섹션(section) 단위로 데이터를 내려보낸다. 각 섹션에는 view.templateType 필드가 있어서, 클라이언트가 어떤 UI로 그릴지 판단할 수 있다.

// v4 응답 구조 (개념적 표현)
{
  "data": {
    "topSections":    [{ "view": { "templateType": "BANNER" }, "data": { ... } }],
    "sortSections":   [{ "view": { "templateType": "SORT" },   "data": { ... } }],
    "filterSections": [{ "view": { "templateType": "FILTER" }, "data": { ... } }],
    "listSections":   [{ "view": { "templateType": "SEARCH_ORGANIC_PRODUCT" }, ... }]
  }
}

이 구조를 클라이언트에서 처리하기 위해 세 가지 설계 결정을 내렸다.

결정 1: Gson 커스텀 역직렬화

섹션마다 data.items의 스키마가 다르다. 배너 섹션은 BannerResponse를, 상품 섹션은 ContentProductListItemResponse를 담고 있다. Gson의 기본 역직렬화로는 이 다형성(polymorphism)을 처리할 수 없다.

SearchSectionTypeAdapter라는 커스텀 JsonDeserializer를 만들어, templateType 값을 기준으로 적절한 매퍼에게 파싱을 위임하는 구조를 설계했다.

결정 2: Mapper 패턴으로 파싱 로직 분리

모든 섹션의 파싱 로직을 TypeAdapter에 넣으면 단일 클래스가 비대해진다. 대신 SearchSectionMapper 추상 클래스를 정의하고, 섹션 타입별로 독립적인 매퍼를 구현했다.

결정 3: KSP로 매퍼 자동 발견

매퍼를 추가할 때마다 TypeAdapter에 수동 등록하면, 기존 구조의 "새 타입 = 여러 파일 수정" 문제가 반복된다. KSP(Kotlin Symbol Processing)로 @SearchSectionBinder 어노테이션이 붙은 매퍼를 빌드 시점에 자동 수집하도록 했다.

v4 API Response
↓
SearchSectionTypeAdapter
(커스텀 역직렬화)
↓
SearchSectionMapper × N
(KSP 자동 발견)
↓
List<SearchPagingModel>

구현 여정

4-1. Data Layer: 동적 파싱

SearchSectionTypeAdapter의 핵심은 deserialize() 메서드다. 4개의 섹션 배열(topSections, sortSections, filterSections, listSections)을 순회하면서, 각 요소의 templateType에 매칭되는 매퍼를 찾아 파싱을 위임한다.

override fun deserialize(
    json: JsonElement?, typeOfT: Type?, context: JsonDeserializationContext?,
): SearchSectionResponse {
    val sections: List<SearchPagingModel> = whenSectionsReceived(json) {
        mapper
            .find { it.templateType == this.getTemplateType() }
            ?.mapping(getItems())
            ?: emptyList()
    }

    val meta: MetaResponse? = whenMetaReceived(json) { /* ... */ }
    val pagination: PaginationResponse? = whenPaginationReceived(json) { /* ... */ }

    return SearchSectionResponse(sections, meta, pagination)
}

여기서 mapper는 모든 SearchSectionMapper 인스턴스의 리스트다. 각 매퍼는 자신이 담당하는 templateType을 선언하고, JSON 요소를 도메인 모델로 변환하는 mapping() 함수를 구현한다.

중요한 설계 포인트는 섹션별 실패 격리다. whenSectionsReceived()는 전체를 runCatching으로 감싸고, 개별 매퍼도 내부에서 runCatching을 사용한다. 하나의 섹션 파싱이 실패해도 나머지 섹션은 정상적으로 표시된다.

4-2. Mapper 패턴: 섹션 타입별 분리

SearchSectionMapper 추상 클래스가 매퍼의 계약을 정의한다.

abstract class SearchSectionMapper {
    abstract val templateType: String
    abstract fun mapping(elements: List<JsonElement>): List<SearchPagingModel>

    protected val gson = Gson()

    protected inline fun <reified T> JsonElement.fromJson(): T {
        return gson.fromJson(this, T::class.java)
    }
}

배너 섹션 매퍼를 예로 보면, 구현이 간결하다.

@SearchSectionBinder
object BannerSectionMapper : SearchSectionMapper() {

    override val templateType: String
        get() = "BANNER"

    override fun mapping(elements: List<JsonElement>): List<SearchPagingModel> {
        return elements
            .runCatching {
                firstOrNull()
                    ?.fromJson<BannerResponse>()
                    ?.toDomainModel()
                    ?.let { listOf(SearchPagingModel.SearchPagingBanner(it)) }
            }.getOrNull()
            ?: emptyList()
    }
}

각 매퍼가 object로 선언되어 싱글톤이며, @SearchSectionBinder 어노테이션으로 KSP에 등록된다. 새 섹션 타입을 추가하려면 이와 같은 매퍼 클래스 하나만 만들면 된다.

이 PR에서 구현한 매퍼는 5종이다.

매퍼templateType출력 모델
SearchProductSectionMapperSEARCH_ORGANIC_PRODUCTSearchPagingProduct
BannerSectionMapperBANNERSearchPagingBanner
KeywordCastingSectionMapperNOTICE_REPLACE_SEARCHSearchKeywordCasting
FilterSectionMapperFILTERSearchPagingFilter
SortSectionMapperSORTSearchPagingSort

4-3. KSP: 빌드 시점 매퍼 수집

매퍼가 5개일 때는 수동 등록도 가능하다. 그러나 Server Driven UI의 본질은 섹션 타입이 계속 늘어난다는 점이다. 매퍼를 추가할 때마다 TypeAdapter의 리스트를 수정해야 한다면, 등록 누락이 곧 파싱 실패로 이어진다.

@SearchSectionBinder 어노테이션과 KSP 프로세서로 이 문제를 해결했다.

// 어노테이션 정의 (annotations/search 모듈)
@Target(AnnotationTarget.CLASS)
annotation class SearchSectionBinder

프로세서는 빌드 시점에 @SearchSectionBinder가 붙은 클래스를 모두 수집하여 SearchSectionMapperFactory를 자동 생성한다.

// KSP 프로세서 핵심 로직
override fun process(resolver: Resolver): List<KSAnnotated> {
    val symbols = resolver
        .getSymbolsWithAnnotation(SearchSectionBinder::class.java.name)
        .filterIsInstance<KSClassDeclaration>()

    // SearchSectionMapperFactory 파일 생성
    val mapperList = symbols
        .map { it.simpleName.asString() }
        .joinToString(separator = ",\n            ")

    // → object SearchSectionMapperFactory {
    //       operator fun invoke(): List<SearchSectionMapper> = listOf(...)
    //   }
    // 가 자동 생성된다.
    return symbols.filterNot { it.validate() }.toList()
}

이로써 SearchSectionTypeAdapter에서 매퍼 리스트를 하드코딩할 필요가 사라졌다.

// TypeAdapter에서의 사용
private val mapper: List<SearchSectionMapper> = SearchSectionMapperFactory()

4-4. UI Layer: 어댑터 통합

Data Layer의 변경이 UI Layer에 파급 효과를 만들었다. v4 응답은 모든 섹션을 List<SearchPagingModel>이라는 단일 스트림으로 내려보낸다. 배너, 필터, 키워드 캐스팅이 더 이상 헤더 영역의 별도 데이터가 아니라, 상품과 동일한 스트림의 일부가 된 것이다.

이 변화로 ProductPagingHeaderAdapter의 존재 이유가 사라졌다. 156줄의 헤더 어댑터를 삭제하고, 모든 섹션 타입을 ProductPagingListAdapter 하나에서 처리하도록 통합했다.

Before (v3)

ConcatAdapter 구성:

ProductPagingHeaderAdapter
  + ProductPagingListAdapter
  = ConcatAdapter

헤더/상품 데이터를 각각 별도로 전달

After (v4)

단일 어댑터:

ProductPagingListAdapter
  (모든 섹션 타입 처리)

PagingData<ContentItemUIModel> 단일 스트림

Domain Layer에서는 SearchPagingModel sealed interface를 확장하여 새 섹션 타입들을 추가했다. SearchKeywordCasting은 검색어 교정 로직을 캡슐화하는 좋은 예다.

data class SearchKeywordCasting(
    private val header: Header,
    private val footer: Footer,
) : SearchPagingModel {

    val keywordCastingType: KeywordCastingType
        get() = KeywordCastingType.create(header.keyword, footer.keyword)

    // footerKeyword가 null이면 대체 제안, 있으면 오타 교정
}

KeywordCastingType은 footerKeyword의 null 여부로 오타 교정(SPELLING_CORRECTION)과 키워드 대체(ALTERNATIVE_SUGGESTION)를 구분한다. 이 로직이 도메인 모델에 캡슐화되어 있으므로, ViewModel과 UI는 타입만 확인하면 된다.

빌드 설정 복잡도가 늘어난 것은 사실이다. annotations/search와 processor/search 두 모듈이 추가되었고, 빌드 변형(variant)별로 KSP 생성 코드의 소스셋을 설정해야 했다. 또한 커스텀 JsonDeserializer는 Gson 기본 역직렬화보다 디버깅이 까다롭다 — JSON 구조가 예상과 다를 때 에러 메시지가 덜 직관적이다.

이 트레이드오프를 감수한 이유는 명확하다. 검색 지면의 실험 속도가 클라이언트 릴리즈 주기에 묶이지 않게 되었고, 새 섹션 타입 추가의 비용이 "매퍼 클래스 하나"로 수렴했다.

남긴 것들

핵심 요약
  • Server Driven UI는 "실험 속도"가 병목일 때 도입하라. 검색처럼 섹션 조합이 자주 바뀌는 지면에 적합하다. 레이아웃이 고정된 화면에는 과잉 설계다.
  • 커스텀 역직렬화에는 반드시 섹션별 실패 격리를 확보하라. runCatching으로 개별 섹션의 파싱 실패가 전체 응답을 무효화하지 않도록 해야 한다. 서버가 새 섹션 타입을 내려보냈는데 클라이언트에 매퍼가 없는 경우에도 나머지 섹션은 정상 표시되어야 한다.
  • KSP는 boilerplate 제거 이상의 가치가 있다. 매퍼 자동 수집은 "등록 누락 = 파싱 실패"라는 위험을 빌드 시점에 제거한다. 새 타입 추가의 진입 장벽을 낮춰서, 팀원 누구나 매퍼 하나만 만들면 새 섹션을 지원할 수 있게 된다.

이 마이그레이션은 90개 파일, 42개 커밋에 걸친 작업이었다. 단순히 API 버전을 올린 것이 아니라, "서버가 UI 구성을 결정한다"는 원칙을 클라이언트 아키텍처 전반에 반영하는 과정이었다.

여러분의 앱에서 "새 UI 타입 = 클라이언트 릴리즈"인 지면을 찾아보라. 그 지면에서 서버 팀이 얼마나 자주 실험을 요청하는지 확인하면, Server Driven UI 도입이 필요한지 판단할 수 있다.

한 가지 주의할 점이 있다. Server Driven UI는 클라이언트의 유연성을 높이는 대신, 서버와 클라이언트 간의 계약(contract)을 더 엄밀하게 관리해야 한다. templateType 문자열 하나의 오타가 파싱 실패로 이어지므로, 서버-클라이언트 간 섹션 타입 정의를 공유하는 메커니즘이 함께 필요하다.
