세 가지 제약

AI 대화형 UI를 기존 커머스 검색에 얹으려고 했을 때, 일반적인 리스트 UI와는 다른 세 가지 제약에 직면했다.

  1. 응답이 다형적이다. AI가 돌려주는 응답은 텍스트일 수도, 상품 비교표일 수도, 추천 상품 목록일 수도 있다. 하나의 응답에 이 모든 블록이 섞여 온다.
  2. 대화가 누적된다. 채팅처럼, 이전 응답 위에 새 응답이 쌓인다. 후속 질문, 재시도, 다른 질문 보기 등 상태 전이가 복잡하다.
  3. 검색 결과 지면을 벗어나면 안 된다. 별도 Activity로 이동하면 돌아왔을 때 대화가 유지되는 것이 어색하다. 같은 키워드로 검색하는 동안에는 대화 맥락이 자연스럽게 이어져야 한다.

세 번째 제약이 BottomSheet라는 컨테이너를 결정했다. 검색 결과 화면 위에 열고 닫을 수 있는 BottomSheet는, 열어 두면 대화가 이어지고 닫아도 다시 열면 이전 내용이 그대로인 것이 자연스럽다.

이 글에서는 이 세 가지 제약을 해결하기 위해 설계한 블록 프로토콜 아키텍처를 해부한다. 도메인의 sealed interface가 계약을 정의하고, 데이터 레이어의 Registry가 다형적 JSON을 역직렬화하고, Resolver가 도메인을 UI 모델로 변환하고, ViewModel이 대화 상태를 관리하는 전체 흐름을 따라간다.

전체 아키텍처 멘탈 모델

먼저 전체 구조를 한눈에 보자. 사용자의 질문이 AI 응답으로 변환되어 화면에 렌더링되기까지의 경로다.

API
Sophie API — JSON { type, data }
블록 타입별로 type 문자열 + data 객체 쌍
↓
Data Layer
TypeAdapter + 8 Mapper Registry
type 문자열로 Mapper를 찾아 data를 도메인 모델로 변환
↓
Domain Layer
AskSophieConversation sealed interface — 12 타입
블록 프로토콜 — 앱 전체가 합의하는 계약
↓
UI Layer
Resolver → UIModel → 9 Block Composables
도메인 12타입 → UIModel + 콜백 주입 → Compose 블록 렌더링

핵심 아이디어를 한 문장으로 요약하면 이렇다. 서버가 보내는 블록 타입을 도메인 모델이 정의하고, 데이터 레이어가 역직렬화하고, UI 레이어가 렌더링한다. 우리는 이 구조를 "블록 프로토콜"이라 부르기로 했다. AI 응답의 각 조각을 독립적인 블록으로 취급하고, 새로운 블록 타입이 추가될 때마다 각 레이어에 하나씩 구현을 추가하는 방식이다.

아래에서 이 다이어그램의 각 레이어를 하나씩 분해한다.

블록 프로토콜: sealed interface가 계약이 되기까지

블록 프로토콜의 출발점은 도메인 레이어의 AskSophieConversation sealed interface다. AI 응답에 포함될 수 있는 모든 블록 타입을 여기서 정의한다.

sealed interface AskSophieConversation {

    data object Loading : AskSophieConversation
    data object Symbol : AskSophieConversation
    data class Error(val question: String, val questionId: String, ...) : AskSophieConversation

    // 텍스트 블록
    data class Intro(val content: String) : AskSophieConversation
    data class Title(val content: String) : AskSophieConversation
    data class Description(val content: String) : AskSophieConversation
    data class Outro(val content: String) : AskSophieConversation

    // 리치 콘텐츠 블록
    data class Comparison(val headers: List<String>, val rows: List<Row>) : AskSophieConversation
    data class Products(val products: List<SearchProduct>) : AskSophieConversation
    data class BulletList(val contents: List<String>) : AskSophieConversation

    // 인터랙션 블록
    data class Summary(val title: String, val questions: List<AskSophieQuestion>) : AskSophieConversation
    data class Suggestion(val contents: List<AskSophieQuestion>) : AskSophieConversation
}

12가지 타입이지만, 역할별로 세 그룹으로 나뉜다.

  • 텍스트 블록 (Intro, Title, Description, Outro) — 단순 텍스트. AI의 설명 문단을 표현한다.
  • 리치 콘텐츠 블록 (Comparison, Products, BulletList) — 구조화된 데이터. 비교표, 상품 카드, 글머리 기호 목록.
  • 인터랙션 블록 (Summary, Suggestion) — 사용자가 후속 질문을 선택할 수 있는 블록.

여기서 눈여겨볼 점이 있다. Loading과 Error는 서버가 보내는 블록이 아니라 클라이언트가 생성하는 UI 전용 상태다. 그런데도 같은 sealed interface에 포함시켰다. UseCase가 응답을 조립할 때 이 타입들을 직접 리스트에 추가하기 때문이다. 도메인 레이어에서 이 타입을 알아야 ViewModel이 대화 리스트에 일관된 타입으로 블록을 쌓을 수 있다.

sealed interface를 사용하면 when 분기에서 새 타입을 추가했을 때 처리하지 않은 곳을 컴파일러가 잡아준다. 새 블록 타입이 추가될 때마다 TypeAdapter, Resolver, BottomSheet에서 컴파일 에러가 발생하고, 이 에러를 따라가면 누락 없이 구현을 완료할 수 있다.

Registry 패턴: 다형적 JSON을 도메인으로

도메인이 계약을 정의했으니, 이제 서버 응답을 이 계약에 맞게 변환해야 한다. Sophie API의 응답은 이런 구조다.

// Sophie API 응답 구조
{
  "answer": [
    { "type": "title",      "data": { "content": "유기농 우유 비교" } },
    { "type": "comparison", "data": { "headers": [...], "rows": [...] } },
    { "type": "products",   "data": { "products": [...] } },
    { "type": "suggestion", "data": { "contents": [...] } }
  ]
}

answer 배열의 각 요소가 type과 data의 쌍이다. 같은 배열 안에 서로 다른 구조의 data가 섞여 있으므로, 일반적인 Gson 자동 역직렬화로는 처리할 수 없다. 이 문제를 해결하기 위해 Registry 패턴을 도입했다.

Mapper 추상 클래스

블록 하나를 변환하는 책임을 가진 AskSophieMapper 추상 클래스를 정의한다.

abstract class AskSophieMapper {
    abstract val type: String                      // "comparison", "title" 등
    abstract fun mapping(data: JsonElement): AskSophieConversation?
    protected val gson = Gson()
}

각 Mapper는 자신이 처리할 수 있는 type 문자열을 선언하고, mapping()에서 JsonElement를 도메인 모델로 변환한다. 예를 들어 ComparisonBlockMapper는 이렇게 생겼다.

object ComparisonBlockMapper : AskSophieMapper() {

    override val type = "comparison"

    override fun mapping(data: JsonElement): AskSophieConversation? {
        return runCatching {
            val obj = data.asJsonObject
            val headers = obj.getAsJsonArray("headers").map { it.asString }
            val rows = obj.getAsJsonArray("rows").map { rowElement ->
                AskSophieConversation.Comparison.Row(
                    label = rowElement.asJsonObject.get("label").asString,
                    values = rowElement.asJsonObject.getAsJsonArray("values").map { it.asString },
                )
            }
            AskSophieConversation.Comparison(headers, rows)
        }.getOrNull()
    }
}

runCatching으로 감싼 이유는 서버 응답이 예상과 다를 때 해당 블록만 건너뛰기 위해서다. 하나의 블록 파싱 실패가 전체 응답을 날리지 않는다.

TypeAdapter: Registry로 디스패치

이제 이 Mapper들을 Registry로 묶는 TypeAdapter를 보자.

class AskSophieConversationTypeAdapter @Inject constructor()
    : JsonDeserializer<AskSophieResponse> {

    private val registry = listOf(
        BulletListMapper,       ComparisonBlockMapper,
        DescriptionBlockMapper, IntroBlockMapper,
        OutroBlockMapper,       ProductsBlockMapper,
        SuggestionsBlockMapper, TitleBlockMapper,
    )

    private fun mapping(element: JsonElement): AskSophieConversation? {
        val type = element.asJsonObject.get("type").asString
        return registry
            .find { mapper -> mapper.type.equals(type, ignoreCase = true) }
            ?.mapping(data = element.asJsonObject.get("data"))
    }
}

동작은 단순하다. JSON 배열을 순회하면서 각 요소의 type을 읽고, registry에서 해당 type을 처리할 수 있는 Mapper를 find로 찾아 위임한다. 매칭되는 Mapper가 없으면 null을 반환하여 해당 블록을 건너뛴다.

새 블록 추가 시 해야 할 일

서버에서 새로운 블록 타입(예: "recipe")이 추가되면:

1. 도메인에 AskSophieConversation.Recipe 추가
2. RecipeBlockMapper object 생성 (type = "recipe")
3. TypeAdapter의 registry 리스트에 한 줄 추가
4. Resolver에 when 분기 추가, Composable 블록 구현

sealed interface 덕분에 4를 빠뜨리면 컴파일 에러가 발생한다.

Resolver 패턴: 도메인에서 UI로의 다리

도메인 모델이 UI에 직접 들어갈 수 없는 이유가 있다. AskSophieConversation.Suggestion에는 "사용자가 질문을 탭했을 때 실행할 콜백"이 없다. 도메인 모델은 순수한 데이터인 반면, UI 모델은 콜백, 스타일, 마진 같은 표현 정보를 품어야 한다.

AskSophieConversationResolver가 이 간극을 메운다.

object AskSophieConversationResolver {

    fun resolve(
        keyword: String,
        questionId: String?,
        conversation: AskSophieConversation,
        isRefreshEnabled: Boolean,
        onNavigate: (SearchNavigator.Destination) -> Unit,
        onAsk: (AskVariant) -> Unit,
        onRefresh: (RefreshSophieUseCase.Param) -> Unit,
    ): AskSophieConversationUIModel = when (conversation) {
        is AskSophieConversation.Loading    -> UIModel.Loading
        is AskSophieConversation.Symbol     -> UIModel.Symbol()
        is AskSophieConversation.Title      -> UIModel.Text(text = it.content, ...)
        is AskSophieConversation.Products   -> UIModel.Products.from(it, onNavigate, onAsk)
        is AskSophieConversation.Suggestion -> UIModel.Suggestion.from(it, onAsk, onRefresh)
        // ... 12가지 타입 모두 처리
    }
}

Resolver가 하는 일은 세 가지다.

  1. 타입 매핑: 도메인의 12 sealed 타입을 UI의 sealed class로 변환한다. Intro, Outro, Description은 모두 UIModel.Text로 매핑되지만, 각각 다른 스타일과 마진을 갖는다.
  2. 콜백 주입: onAsk, onNavigate, onRefresh 같은 콜백을 UIModel에 주입한다. 이 콜백은 ViewModel에서 제공되며, 사용자 인터랙션을 ViewModel로 전달하는 역할을 한다.
  3. 컨텍스트 전파: keyword, questionId, isRefreshEnabled 같은 현재 대화 맥락을 UI 모델에 전달한다.

처음에 Resolver는 @Composable 함수였다. 도메인 타입에 따라 UI 스타일(KurlyTheme.typography 등)을 직접 참조했기 때문이다. 그런데 Composable 컨텍스트 밖에서는 호출할 수 없으므로, 단위 테스트를 작성할 수 없었다.

해결책은 UI 스타일 참조를 @Composable 람다로 지연시키는 것이었다.

// UIModel.Text의 스타일 프로퍼티를 람다로 선언
data class Text(
    val text: String,
    val textColor: @Composable () -> Color,      // 렌더링 시점에 평가
    val textStyle: @Composable () -> TextStyle,  // Compose 컨텍스트에 의존
)

이렇게 하면 Resolver 자체는 object의 일반 함수로 남아 테스트가 가능하고, 테마 의존성은 Composable이 실제로 렌더링되는 시점에 해소된다.

대화 상태 머신: ViewModel이 대화를 관리하는 법

블록 프로토콜의 마지막 조각은 ViewModel이다. AskSophieBottomSheetViewModel은 대화를 List<UIModel>로 관리한다. 새 블록이 오면 리스트에 추가하고, 재시도하면 일부를 제거한 뒤 다시 추가한다.

ask 흐름

사용자가 질문을 선택하면 onAsk()가 실행된다. 리스트를 어떻게 조작하는지 따라가 보자.

1
이전 Suggestion 비활성화
disabledSuggestion()
↓
2
질문 텍스트를 우측 정렬 말풍선으로 추가
generateQuestion(variant.question)
↓
3
Loading 블록 추가 (최소 2초 유지)
AskSophieConversationUIModel.Loading
↓
4
API 호출 & 응답 도착
askSophieUseCase.execute(variant)
↓
5
Loading 제거, Resolver로 변환한 블록들을 리스트에 추가
AskSophieConversationResolver.resolve(...)

핵심 코드를 보자.

private fun onAsk(variant: AskVariant) {
    if (!isIdle()) return                          // 동시 요청 방어
    job = viewModelScope.launch {
        // Step 1-3: Suggestion 비활성화 + 질문 + Loading
        updateState { prev ->
            val conversations = buildList {
                addAll(prev.conversations.dropTailFor(variant))
                add(prev.conversations.disabledSuggestion())
                add(UIModel.generateQuestion(variant.question))
                add(UIModel.Loading)
            }.filterNotNull()
            prev.copy(conversations = conversations)
        }

        // Step 4: API 호출 (최소 로딩 시간 보장)
        val result = await(UIModel.Loading.duration) {
            askSophieUseCase.execute(variant)
        }

        // Step 5: Loading 제거, 응답 블록 추가
        updateState { prev ->
            val newItems = Resolver.resolve(...)
            prev.copy(conversations = prev.conversations
                .filterNot { it is UIModel.Loading } + newItems)
        }
    }
}

await: 최소 로딩 지속시간 보장

await()는 API 응답이 너무 빨리 오더라도 Loading 애니메이션이 최소 시간 동안 유지되도록 하는 패턴이다.

private suspend fun <T> await(minDurationMillis: Long, block: suspend () -> T): T {
    val startedAt = System.currentTimeMillis()
    val result = block()
    val remaining = (minDurationMillis - (System.currentTimeMillis() - startedAt))
        .coerceAtLeast(0L)
    delay(remaining)
    return result
}

Loading 블록에는 "금방 알려드릴게요", "잠시만 기다려주세요!" 두 문장이 순차적으로 표시되며, 이 전환 애니메이션에 최소 2초가 필요하다. API가 500ms 만에 응답하면 delay(1500)으로 남은 시간을 채운다.

refresh: 질문 메모리와 "다른 질문 보기"

Suggestion 블록 하단에는 "다른 질문 보기" 버튼이 있다. 이 버튼을 누르면 서버에 새 질문 목록을 요청하는데, 이미 보여준 질문이 다시 나오면 안 된다. 이 제약을 데이터 레이어의 SearchAskSophieQuestionMemory가 해결한다.

class SearchAskSophieQuestionMemory @Inject constructor() {
    private val cache = mutableSetOf<String>()    // 이미 보여준 질문 ID

    fun consume(questionIds: List<String>): Set<String> {
        cache.addAll(questionIds)
        return cache                              // 서버에 exclude 목록으로 전달
    }
}

Refresh API 호출 시 이 캐시를 questionIdsToExclude로 전달하여, 서버가 이미 노출된 질문을 제외하고 새 질문을 반환하도록 한다. 세션(같은 키워드 검색) 동안 캐시가 누적되므로, 사용자가 "다른 질문 보기"를 반복해도 중복이 발생하지 않는다.

retry: 마지막 질문부터 다시

에러 발생 시 재시도하면, 마지막 질문과 그에 대한 응답을 제거한 뒤 같은 질문을 다시 보내야 한다. calculateDropCount()가 이 역할을 한다.

private fun List<UIModel>.calculateDropCount(variant: AskVariant): Int {
    return when (variant) {
        is AskVariant.Retry -> {
            val lastQ = this.filterIsInstance<UIModel.Text>()
                .lastOrNull { !it.isLeft }    // 우측 정렬 = 사용자 질문
            if (lastQ != null) this.size - this.indexOf(lastQ)
            else 1
        }
        else -> 1                        // 일반 질문: 마지막 Suggestion만 제거
    }
}

동시성 방어: isIdle

사용자가 빠르게 질문을 연타하는 경우를 방어한다. job 필드가 현재 진행 중인 코루틴을 추적하고, isIdle()이 false면 새 요청을 무시한다.

private var job: Job? = null

fun isIdle(): Boolean = job == null || job?.isActive == false

끝까지 따라가기: 질문 한 번의 여정

지금까지 분해한 컴포넌트들이 실제로 어떻게 협력하는지, 사용자가 "이 상품과 비슷한 상품은?"이라는 질문을 탭하는 시나리오를 끝까지 추적해 보자.

그 전에, BottomSheet에 진입하는 경로가 두 가지임을 짚어두자.

두 가지 진입 경로

Intro — 검색 결과 상단 AI 섹션에서 진입. 서버가 미리 생성한 추천 질문 목록(Summary)을 표시하고, 사용자가 질문을 선택하면 API 호출.
Conversation — 플로팅 버튼 또는 상품 카드의 "더 알아보기"에서 진입. 즉시 AskVariant를 생성하여 API 호출을 시작. 상품 수준 질문은 AskVariant.Product를 사용하며, questionId가 "learn_more_$productNo"로 자동 생성된다.

아래 파이프라인은 두 번째 경로(Conversation)의 후속 질문 시나리오를 추적한다.

마지막 단계에서 눈여겨볼 것이 스태거 애니메이션이다. 새로 추가된 블록들은 동시에 나타나는 게 아니라, 250ms 간격으로 순차적으로 fade-in + slide-up된다. 이전 대화까지의 블록은 staggerBaseIndex로 제외하고, 그 이후 블록만 순차 등장시킨다. AI가 "생각하고 있다"는 인상을 시각적으로 전달하는 장치다.

이 경로에서 응답에 Comparison 블록이 포함되어 있다면, ComparisonTableBlock이 렌더링된다. 이 컴포넌트는 Compose의 Layout {}을 직접 구현하여 테이블을 그린다.

ComparisonTable: 왜 custom Layout인가

비교표의 요구사항은 세 가지였다. 첫 번째 열(항목명)은 내용에 맞게 wrap되어야 하고, 나머지 열(비교 대상)은 남은 공간을 균등하게 나눠야 하며, 같은 행의 모든 셀은 높이가 같아야 한다.

처음에는 IntrinsicSize.Max를 적용한 Row/Column 조합으로 시도했다. 그런데 첫 번째 열의 너비를 "전체 행에서 가장 넓은 항목명"에 맞추려면 행 간 intrinsic width를 조율해야 한다. Compose의 intrinsic measurement만으로는 이 조율이 자연스럽지 않았다.

결국 Layout {}을 직접 구현했다. 핵심은 2-pass 접근이다.

Layout(content, modifier) { measurables, constraints ->
    // Pass 1: 열 너비 확정
    val labelColWidth = (0 until rowCount).maxOf { row ->
        measurables[row * columnCount].maxIntrinsicWidth(Int.MAX_VALUE)
    }
    val dataColWidth =
        (constraints.maxWidth - labelColWidth - dividers) / (columnCount - 1)

    // Pass 2: 행 높이 = 해당 행 셀 중 최대값
    val rowHeights = IntArray(rowCount) { row ->
        (0 until columnCount).maxOf { col ->
            measurables[row * columnCount + col].maxIntrinsicHeight(colWidths[col])
        }
    }
    // 각 셀을 고정 크기로 measure & place
}

전체 셀의 intrinsic 크기를 먼저 수집하고, 열 너비와 행 높이를 확정한 뒤, 각 셀을 고정 크기로 measure한다. 셀 사이 1dp 간격은 부모 배경색(Gray200)이 비쳐서 구분선 역할을 한다.

설계 한계와 대안

이 아키텍처가 모든 상황에 최적인 것은 아니다.

Gson TypeAdapter의 한계. 프로젝트가 Gson 기반이므로 자연스러운 선택이었지만, 타입 안전성 측면에서 Kotlin Serialization의 @Polymorphic serializer가 우위에 있다. Kotlin Serialization은 컴파일 타임에 직렬화 코드를 생성하므로, 런타임에 type 문자열을 수동으로 매칭하는 현재 방식보다 오류 가능성이 낮다. 프로젝트 전체의 직렬화 라이브러리가 전환되면 Registry 패턴 자체가 불필요해질 수 있다.

Resolver의 when 분기 증가. 블록 타입이 12개일 때는 관리 가능하지만, 20개, 30개로 늘어나면 Resolver가 비대해진다. 대안으로, 각 UIModel의 companion에 from(domain) 패턴을 두어 변환 책임을 분산하는 방식이 있다. 현재도 Products.from(), Suggestion.from() 등 일부 타입에는 이 패턴이 적용되어 있다.

List 기반 상태 관리의 취약성. 대화 상태를 List로 관리하고 dropLast, indexOf 등 인덱스 기반으로 조작한다. 대화 흐름이 복잡해지면 의미 기반 조작이 인덱스 연산으로 번역되는 과정에서 버그 표면이 넓어진다. 각 대화 턴에 ID를 부여하고 트리 구조로 관리하는 방식이 더 안전할 수 있다.

핵심 설계 원칙

Ask Sophie의 블록 프로토콜 아키텍처를 관통하는 원칙을 정리한다.

TAKEAWAYS
  • sealed interface를 계약으로. AI 응답의 가능한 모든 블록 타입을 도메인 레이어에서 선언한다. 새 타입 추가 시 컴파일러가 누락된 처리를 잡아준다.
  • Registry 패턴으로 다형적 역직렬화. type 문자열 → Mapper 디스패치로, 각 블록의 파싱 로직을 독립적인 object로 캡슐화한다. 하나의 블록 파싱 실패가 전체 응답을 날리지 않는다.
  • Resolver로 레이어 간극 메우기. 순수 도메인 모델에 UI 콜백과 스타일을 주입하는 중간 계층. @Composable 의존성을 람다로 지연시키면 테스트가 가능해진다.
  • 리스트가 곧 대화 상태. 대화를 블록의 리스트로 모델링하면 LazyColumn과 1:1로 매핑된다. 단, 인덱스 기반 조작은 의미 기반 추상화로 감싸야 한다.

이 패턴은 Ask Sophie에 한정되지 않는다. 서버가 다형적 콘텐츠 블록을 보내고, 클라이언트가 이를 해석하여 각 블록 타입에 맞는 UI로 렌더링하는 구조—이른바 Server-Driven UI의 변형—는 AI 대화뿐 아니라 콘텐츠 피드, 동적 폼, 커스텀 페이지 등 다양한 영역에 적용할 수 있다.

블록 프로토콜의 확장력은 sealed interface의 컴파일 타임 안전성에 달려 있다. 새 블록을 추가할 때 "어디를 수정해야 하지?"라고 고민할 필요가 없다. 컴파일러가 알려준다.
