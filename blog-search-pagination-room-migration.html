왜 메모리 캐시로는 안 되었나

검색 결과 페이지네이션에는 단순히 상품을 나열하는 것 이상의 요구사항이 있었다. 페이지네이션 사이사이에 연관 상품, 배너, 구매 이력 등 다양한 섹션을 INSERT해야 했다. 문제는 이 삽입을 메모리 캐시에서 관리하는 것이었다.

기존에는 검색 결과를 Map<Int, List<SearchPagingModel>> 형태로 메모리에 저장했다. 페이지 번호가 키이고, 해당 페이지의 모든 섹션 데이터가 값이었다. 여기에 아이템을 삽입하려면 페이지 경계를 넘나드는 위치 계산, 삽입 후 인덱스 보정, 각 섹션의 메타정보(sectionId, sectionPosition, sectionArea) 관리를 모두 메모리에서 직접 수행해야 했다.

이 부수 정보를 앱 코드에서 하나하나 관리하다 보니 누락이 발생했다. 실제로 일부 케이스에서 삽입된 섹션의 위치 정보가 정확하지 않아 정상 동작하지 않는 경우가 발견되었다.

페이지네이션 데이터와 부수 정보의 관리를 앱 코드가 아닌 Room에 위임하면, 삽입·정렬·조회를 SQL이 책임진다. 이것이 마이그레이션의 핵심 동기였다.

기존 구조: 메모리 캐시 기반 Pagination

마이그레이션 대상을 이해하려면 기존 구조를 먼저 살펴야 한다. 검색 Pagination은 세 개의 핵심 클래스로 구성되어 있었다.

// 메모리 캐시: 페이지 번호 → 검색 결과 리스트
class SearchProductCachePool {
    private val cache = ConcurrentHashMap<Int, List<SearchPagingModel>>()
    private val pageKeys = ConcurrentHashMap<String, PageKeySource>()

    fun put(page: Int, data: List<SearchPagingModel>) { /* ... */ }
    fun get(page: Int): List<SearchPagingModel>? { /* ... */ }
    fun clear() { cache.clear(); pageKeys.clear() }
}

SearchProductCachePool은 148줄의 클래스로, API 응답을 도메인 모델(SearchPagingModel)로 변환한 뒤 페이지 번호를 키로 메모리에 저장했다. SearchProductRemoteMediator가 API를 호출하여 이 캐시에 쓰고, SearchProductCacheSource라는 커스텀 PagingSource가 캐시에서 읽었다.

이 구조에는 한 가지 추가적인 문제가 있었다. API 응답을 역직렬화하는 시점(SearchSectionTypeAdapter)에서 바로 도메인 모델로 변환했기 때문에, 원본 JSON이 보존되지 않았다. 캐시를 디스크로 옮기려면 도메인 모델 전체를 직렬화해야 하는데, SearchPagingModel은 sealed class 계층 구조를 포함하고 있어 직렬화가 단순하지 않았다.

핵심 발상의 전환은 "변환 시점의 지연"이다. 원본 JSON을 Room에 저장하고, 도메인 모델로의 변환은 UI 바인딩 시점으로 미룬다. 이렇게 하면 Room이 삽입·정렬·페이지 관리를 SQL로 처리하고, 앱 코드는 변환에만 집중할 수 있다.

Room Entity 설계

Room 캐시의 설계에서 결정해야 할 것은 세 가지였다. 무엇을 저장할 것인가, 어떤 순서로 정렬할 것인가, 그리고 페이지 키를 어떻게 관리할 것인가.

SearchSectionEntity: 원본 보존 전략

검색 결과의 각 섹션(상품 리스트, 배너, 테마 등)을 하나의 Row로 저장한다. 핵심은 payload 필드에 원본 JSON 문자열을 그대로 보존하는 것이다.

@Entity(tableName = "search_sections")
data class SearchSectionEntity(
    @PrimaryKey(autoGenerate = true)
    val id: Int = 0,
    val payload: String,          // 원본 JSON 문자열
    val actualKeyword: String?,
    val orderKey: Double,         // 정렬 키
    val sectionArea: SearchSectionArea,
    val sectionAreaType: String,
    val sectionId: String,
    val sectionCode: String,
)

도메인 모델 대신 원본 JSON을 저장하면 두 가지 이점이 있다. 첫째, SearchPagingModel의 sealed class 계층 전체를 직렬화할 필요가 없다. 둘째, Mapper 로직이 변경되어도 캐시를 무효화하지 않아도 된다.

orderKey: 페이지 간 안정 정렬

여러 페이지의 섹션을 하나의 테이블에 저장할 때, 페이지 내 순서와 페이지 간 순서를 모두 보존해야 한다. 이를 위해 orderKey를 다음과 같이 설계했다.

private fun orderKeyOf(page: Int, indexInPage: Int): Double {
    return page.toLong() * PAGE_STRIDE + indexInPage.toDouble()
}

private const val PAGE_STRIDE = 1_000_000L

1페이지의 3번째 아이템은 1,000,003, 2페이지의 1번째 아이템은 2,000,001이 된다. 한 페이지에 100만 개 미만의 아이템이 있다는 전제하에, 페이지 간 삽입이나 재정렬 없이 ORDER BY orderKey ASC만으로 올바른 순서를 보장한다.

SearchPageKeyEntity: 페이지 키 분리

@Entity(tableName = "search_page_keys")
data class SearchPageKeyEntity(
    @PrimaryKey
    val keyword: String,
    val prevKey: Int?,
    val nextKey: Int?,
)

RemoteMediator가 다음 페이지를 판단하려면 현재 키워드의 이전/다음 페이지 번호가 필요하다. 이를 섹션 테이블과 분리하여 search_page_keys 테이블에 저장했다. 키워드 변경 시 두 테이블을 함께 초기화하는 트랜잭션(SearchProductTransaction)으로 일관성을 유지한다.

까다로웠던 부분

Room Entity 설계는 비교적 직관적이었다. 예상보다 시간이 걸린 것은 세 가지 부수적인 문제였다.

Challenge 1: sealed class의 Room 저장

SearchSectionArea는 sealed class다. Top, List, None 세 가지 하위 타입이 있고, 각각 다른 마진 값을 갖는다. Room의 TypeConverter는 단일 타입 변환만 지원하므로, 다형성을 보존하려면 별도의 직렬화 전략이 필요했다.

해결 방법은 SerializationSnapshot이라는 래퍼를 도입하는 것이었다.

data class SerializationSnapshot(
    val className: String,       // "Top", "List", "None"
    val jsonElement: JsonElement?  // 실제 데이터
)

직렬화 시 클래스 이름과 JSON 데이터를 함께 저장하고, 역직렬화 시 클래스 이름으로 분기하여 올바른 하위 타입을 복원한다. SearchSectionAreaTypeAdapter가 이 변환을 담당한다.

// 역직렬화: className으로 분기
when (snapshot.className) {
    SearchSectionArea.Top::class.simpleName ->
        context.deserializeOrNull(snapshot.jsonElement, Top::class.java)
    SearchSectionArea.List::class.simpleName ->
        context.deserializeOrNull(snapshot.jsonElement, List::class.java)
    else -> SearchSectionArea.None
}

이 패턴은 Room에 sealed class를 저장해야 하는 다른 상황에도 재사용할 수 있다. 핵심은 타입 정보를 데이터와 함께 저장하는 것이다.

Challenge 2: sectionPosition을 데이터 생성 시점에 확정

기존에는 sectionPosition을 Adapter의 bind() 시점에 매번 계산했다. "이 아이템이 같은 섹션에서 몇 번째인지"를 알려면, 전체 스냅샷을 순회하여 같은 섹션 이름을 가진 아이템을 모은 뒤 인덱스를 찾아야 했다.

// Before: bind()할 때마다 전체 스냅샷 순회
private fun getSectionPosition(target: SearchSectionItemUIModel): Int? {
    return snapshot().items
        .filter { it.sectionArea.getName() == target.sectionArea.getName() }
        .groupBy { it.sectionId }
        .keys.toList()
        .findIndex { it == target.sectionId }
}

Room 마이그레이션과 함께 이 값을 API 응답을 파싱하는 시점에 한 번만 계산하도록 변경했다. SearchSectionTypeAdapter가 각 섹션의 sectionId와 sectionPosition을 확정하고, 이 값이 Entity → UI 모델까지 그대로 전달된다.

// After: 파싱 시점에 한 번만 계산하여 Entity에 저장
SearchSectionEntity(
    payload = this.toString(),
    orderKey = orderKeyOf(page, it.index),
    sectionId = it.sectionId,       // 파싱 시점에 확정
    sectionCode = it.sectionCode,
    // ...
)

결과적으로 ViewHolder는 더 이상 위치를 계산할 필요가 없다. UI 모델이 이미 자신의 위치를 알고 있으므로, bind()의 파라미터가 단순해졌다.

Before

bind(item, getSectionPosition(item))

After

bind(item) // item.sectionPosition 내장

Challenge 3: peek() vs getItem()

PagingDataAdapter의 getItem(position)은 해당 위치의 데이터를 요청하면서 동시에 페이지 로드를 트리거한다. getItemViewType()에서 이 메서드를 호출하면, ViewType을 확인하는 것만으로도 불필요한 페이지 프리페치가 발생할 수 있다.

peek(position)은 이미 로드된 데이터만 반환하고 새로운 로드를 트리거하지 않는다. getItemViewType()과 getSpanSize()처럼 데이터 관찰만 필요한 메서드에서는 peek()이 적합하다.

// Before: 로드를 트리거할 수 있음
override fun getItemViewType(position: Int): Int {
    return when (getItem(position)) { /* ... */ }
}

// After: 로드를 트리거하지 않음
override fun getItemViewType(position: Int): Int {
    return when (peek(position)) { /* ... */ }
}

전체 코드량은 447줄 증가했다. 특히 SearchSectionTypeAdapter가 45줄에서 270줄로 크게 늘었는데, 기존에 각 Mapper에 분산되어 있던 섹션 분해 로직(resolveTopSections, resolveListSections 등)과 타이틀/아이템 분리 로직이 TypeAdapter로 집중되었기 때문이다.

코드량 증가는 트레이드오프다. 대신 얻은 것은 캐시의 프로세스 생존성, 바인딩 시점의 O(1) 위치 접근, 그리고 안정적인 아이템 식별이다. 우리 팀에서는 이 교환이 합리적이라고 판단했다.

부수적 개선

Room 마이그레이션과 함께 진행한 정리 작업도 있다.

  • extensition → extension 패키지명 오타 수정
  • StickerTypeAdapter → StickerResponseTypeAdapter 네이밍 명확화
  • Sticker 도메인 모델의 type → stickerType (Room 예약어 충돌 방지)
  • SpaceItemDecoration에서 GridLayoutManager.LayoutParams.spanIndex 직접 참조 (ViewHolder 타입 체크 제거)

마무리

핵심 요약
  • 원본 JSON을 저장하라. 도메인 모델의 sealed class 계층을 직렬화하는 대신, 원본 JSON을 Room에 보존하고 변환을 UI 바인딩 시점으로 미루면 직렬화 복잡도를 크게 줄일 수 있다.
  • 위치 정보를 모델에 내재화하라. sectionPosition처럼 런타임에 반복 계산하는 값은 데이터 생성 시점에 한 번 계산하여 모델에 포함시키면 Adapter의 bind()가 단순해지고 O(n) 스캔이 사라진다.
  • peek()과 getItem()을 구분하라. PagingDataAdapter에서 데이터 관찰만 필요한 곳(getItemViewType, getSpanSize)에서는 peek()을 사용하여 불필요한 페이지 로드 트리거를 방지하라.

한 가지 주의할 점이 있다. 모든 Pagination 캐시를 Room으로 옮길 필요는 없다. 데이터가 수십 개 이하이고, 프로세스 재생성 시 처음부터 다시 로드해도 사용자 경험에 영향이 없다면 메모리 캐시가 더 단순한 선택이다. Room은 "캐시가 프로세스를 넘어 생존해야 하는가?"라는 질문에 "예"라고 답할 때 선택하는 것이 합리적이다.
