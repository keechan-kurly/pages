네비게이션 구조가 어떻게 바뀌었나

이번 PR의 중심은 호출 코드를 조금 손본 것이 아니라, 앱이 화면 이동을 표현하고 실행하는 방식 자체를 바꾼 데 있습니다. 즉 “어디로 이동할지”와 “그 이동을 누가 실행할지”를 더 단순한 구조로 다시 나눴습니다.

Before에서는 화면 진입마다 전용 Delegator 또는 세분화된 nav delegator가 필요했습니다. After에서는 호출부가 Destination.Screen.* 을 선언하고, navigator 구현이 resolver map에서 적절한 ScreenResolver 를 찾아 실행합니다. 이 차이가 이 PR 전체를 관통합니다.

지금의 화면 이동은 이렇게 동작합니다

핵심 관계를 쉽게 풀면 이렇습니다. 호출부는 어디로 갈지만 Destination 으로 선언합니다. Resolver 는 그 목적지를 받아서 실제로 무슨 Intent를 만들지, 또는 어떤 Activity/Fragment를 열지를 결정합니다. IntentGenerator 는 resolver가 필요할 때 쓰는 하위 도구입니다.

한 줄 요약: Destination 은 "이동 요청서", Resolver 는 "실행 담당자", IntentGenerator 는 "실행에 필요한 intent 조립 도구"에 가깝습니다.
1
호출부
Feature, Activity, Fragment가 이동을 요청합니다. 이제는 이동 방법보다 어디로 갈지를 먼저 말합니다.
예: 상품 상세로 이동, 홈으로 이동
2
목적지 선언
Destination.Screen.ProductDetail(...) 같은 값 객체로 이동 의도를 표현합니다. 필요한 파라미터도 이 단계에 담깁니다.
무엇을 열지와 필요한 값이 여기서 정리됩니다.
3
KurlyNavigator 선택
받은 목적지 타입을 보고, 이 요청을 처리할 맞는 Resolver 를 찾습니다.
여기서는 실행하지 않고, 담당자를 연결합니다.
4
Resolver 해석
실제 실행 방법을 결정합니다. Activity를 열지, deep link URI를 만들지, 어떤 인자를 붙일지 여기서 정합니다.
필요하면 이 단계에서 IntentGenerator 같은 도구를 사용합니다.
5
실제 실행
최종적으로 Intent 실행, Activity 시작, Fragment 진입 같은 실제 화면 이동이 일어납니다.
사용자는 여기서 실제 화면 전환을 보게 됩니다.
Destination
이동 의도를 데이터로 표현합니다. 예를 들어 Home, Cart, ProductDetail(productNo, referrer) 같은 형태입니다. 호출부는 화면을 여는 방법이 아니라 목적지만 말합니다.
Resolver
해당 목적지를 실제 앱 실행으로 바꿉니다. 어떤 Activity를 열지, 어떤 인자를 붙일지, deep link URI를 쓸지 등을 여기서 결정합니다.
IntentGenerator
새 구조의 중심이 아니라, 일부 resolver가 필요할 때 쓰는 보조 도구입니다. 즉 항상 앞에 나오는 개념이 아니라 실행 세부 구현에 가깝습니다.
기존 구조가 복잡했던 이유
  • 호출부가 어떤 delegator를 써야 하는지 알아야 했습니다.
  • 네비게이션 로직이 여러 우회 객체에 흩어져 있었습니다.
  • Intent 생성 도구와 화면 실행 책임이 섞여 보였습니다.
지금 더 읽기 쉬운 이유
  • 호출부는 목적지만 선언하면 됩니다.
  • Resolver가 실행 책임을 맡는다는 규칙이 명확합니다.
  • IntentGenerator는 필요할 때만 resolver 내부에서 보조적으로 등장합니다.

딥링크 표현도 함께 바뀌었습니다

이번 PR은 호출 API만 바꾼 것이 아니라, 목적지 모델이 deep link 경계를 더 많이 품도록 확장합니다. 특히 ProductDetail 과 Collection 변화가 이 점을 드러냅니다.

ProductDetail 목적지
목적지 모델이 referrer를 직접 포함
+data class ProductDetail(val productNo: Long, val referrerEventName: String? = null)
+fromUri(...) reads query param: referrer
+toUri(...) writes query param: referrer
productNo = required referrer = optional URI <-> model symmetry
Collection 목적지
code가 nullable로 확장
-val code: String
+val code: String?
Collection 목적지 표현을 넓혀서, 링크 생성기와 resolver가 더 유연한 입력 상태를 처리할 수 있게 바뀝니다. 즉 deep link 모델이 실제 호출부의 다양성을 더 직접 반영합니다.

이 변경으로 무엇이 쉬워졌나

이제 더 쉽게 할 수 있는 일
  • 새 화면 이동 코드를 읽을 때, 예전처럼 여러 delegator 타입을 추적하기보다 Destination 하나를 먼저 보면 됩니다.
  • 새 목적지를 추가할 때, 어디를 고쳐야 하는지가 더 분명합니다. 목적지 모델을 만들고 resolver 등록을 연결하면 됩니다.
  • 링크 파라미터와 화면 진입 정보가 Destination 모델에 가까워져서, deep link 규칙을 추적하기 쉬워집니다.
  • 결과적으로 네비게이션 관련 수정이 들어왔을 때 “무슨 우회 객체를 써야 하지?”보다 “어떤 목적지로 이동하지?”를 먼저 생각하게 됩니다.
