KMA-3701 · Kurly Android

ComposeView 높이 갱신 이슈

GridLayoutManager가 고정 크기 제약을 전달하면, AndroidComposeView가 invalidate()를 호출하여 높이 갱신이 누락된다

배경 지식

invalidate vs requestLayout 차이점

invalidate()
View의 그리기(draw)만 재요청한다. 크기와 위치는 변경되지 않는다.

→
layout

버그 메커니즘

ComposeView 내부 측정 흐름

1onBindViewHolder()
ViewHolder가 재활용되면서 composeView.setContent { ... } 호출
2ComposeView.onMeasure() → AndroidComposeView.onMeasure()
아직 Content() 호출이 발생하지 않은 시점에 크기를 측정한다. 부모로부터 전달받은 MeasureSpec을 기반으로 root constraints를 갱신한다.
3updateRootConstraints() → markMeasurePending()
이전 Composition 결과를 기반으로 잘못된 크기가 측정될 수 있다. 측정 대기(Pending) 상태로 설정.
4Content() 호출 → dispatchDraw() → Pending 해소
새로운 Composition이 완료되고, Pending 상태가 해소되면서 크기 재측정 로직이 실행된다.
5scheduleMeasureAndLayout()
핵심 분기: childSizeCanAffectParentSize()를 확인하여 부모 크기에 영향을 줄 수 있는지 판정한다.
✓ true — 부모 크기 영향 가능
requestLayout()
measure → layout → draw 전체 수행
높이가 정상 갱신된다
✗ false — 부모 크기 영향 불가
invalidate()
draw만 수행, 크기 재측정 없음
높이가 갱신되지 않는다
핵심 인사이트
childSizeCanAffectParentSize()는 hasFixedInnerContentConstraints를 검사한다. 이 값은 AndroidComposeView.onMeasure() 시점에 갱신된 Constraints를 참조하는데, 아직 새 Composition이 반영되기 전의 제약이다. 즉, 이전 콘텐츠 기준의 제약으로 판단을 내리는 것이 문제의 근본 원인이다.

4
invalidate() 호출
높이 갱신 실패 — draw만 수행
GridLayoutManager 동작 원리
spanCount ≥ 2일 경우, 같은 행의 여러 아이템 중 최대 높이를 기준으로 모든 아이템의 높이를 동일하게 설정한다. 이로 인해 MeasureSpec이 EXACTLY 또는 AT_MOST로 전달되어 고정 제약이 설정된다. 구글은 이 문제를 인지하고 있으나 별도 수정 계획은 없다고 밝혔다.
수정 방법

수동 requestLayout() 호출

✓
PR #4211 — SearchProductViewHolder
composeView.setContent { ProductScreen( item = item, sectionPosition = sectionPosition ?: Int.MIN_VALUE, productPosition = productPosition ?: Int.MIN_VALUE ) } // GridLayoutManager + ComposeView 조합으로 구현 시 // ViewHolder 높이가 갱신되지 않는 이슈로 인해 추가 composeView.getChildAt(0)?.requestLayout()
composeView.getChildAt(0)은 내부의 AndroidComposeView를 가리킨다. setContent 직후에 수동으로 requestLayout()을 호출하면, scheduleMeasureAndLayout()의 잘못된 분기 판정과 무관하게 measure → layout → draw 전체 사이클이 강제 실행되어 높이가 정상 갱신된다.
Cognitive Rationale — 이 시각화가 이렇게 구성된 이유
Pattern
PAT-024 — Data Flow Diagram (커스텀 변형)
시스템의 동작/행위를 설명하는 것이 핵심이므로 클래스 계층 구조보다 메서드 호출 체인 + 결정 분기를 시각화했다. 변경의 본질이 "순차 흐름에서 잘못된 분기를 타는 메커니즘"이므로 PAT-024(데이터 흐름도)를 기반으로, Step 5에 Decision Gate 패턴을 삽입하여 두 경로의 결과를 즉각 대비시켰다.
