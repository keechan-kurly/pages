이 3가지를 각각 별도 PR로 해결했다.

워크플로우 실행 정책 개선 (#7235)

build_for_develop.yml에 check-changes job을 추가하여, 테스트 코드만 변경된 PR에서는 릴리즈 빌드를 스킵하도록 변경.

핵심 변경

  • check-changes job: PR 변경 파일 목록에서 src/test/, src/androidTest/ 경로를 제외한 파일이 있는지 확인
  • should_build output으로 다음 release-build job을 조건부 실행
  • concurrency 설정을 release-build job 레벨로 이동 — 프로덕션 코드 변경 시에만 이전 빌드를 취소하고, 테스트 코드 변경 시에는 불필요한 취소를 방지

check-changes job

# build_for_develop.yml
check-changes:
  name: "Check changed files"
  runs-on: ubuntu-latest
  outputs:
    should_build: ${{ steps.check.outputs.should_build }}
  steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0
    - id: check
      run: |
        if [ "${{ github.event.action }}" = "synchronize" ] && \
           git cat-file -e ${{ github.event.before }} 2>/dev/null; then
          # 일반 push: 최신 push의 변경 파일만 확인
          CHANGED=$(git diff --name-only \
            ${{ github.event.before }} ${{ github.event.after }})
        else
          # PR 생성, reopen, force push: base 브랜치 대비 전체 diff
          CHANGED=$(git diff --name-only \
            origin/${{ github.event.pull_request.base.ref }}...HEAD)
        fi

        NON_TEST=$(echo "$CHANGED" \
          | grep -v '^.*/src/test/.*' \
          | grep -v '^.*/src/androidTest/.*' || true)

        if [ -z "$NON_TEST" ]; then
          echo "should_build=false" >> $GITHUB_OUTPUT
        else
          echo "should_build=true" >> $GITHUB_OUTPUT
        fi

release-build job 조건부 실행

release-build:
  needs: check-changes
  if: needs.check-changes.outputs.should_build == 'true'

  # concurrency를 job 레벨로 이동
  concurrency:
    group: debug_build-${{ github.head_ref }}
    cancel-in-progress: true
synchronize 이벤트에서 github.event.before 커밋이 존재하면 incremental diff, 없으면 (force push 등) base 대비 전체 diff를 사용한다.

모든 PR → 빌드 + 테스트 실행

After

check-changes
테스트만 변경? → 스킵
프로덕션 변경? → 빌드

실패 유형 자동 분류 (#7271)

unit_test_on_pr.yml을 2-Phase 구조로 재설계하여, 실패 원인을 자동으로 분류하고 PR 코멘트 + Slack 알림에 반영.

2-Phase 실행 전략

Before

testStoreRelease 단일 실행
실패 시 동일한 에러 메시지

After

Phase 1: 변경 모듈만 컴파일 체크
Phase 2: 컴파일 성공 시 전체 테스트

Phase 1: 변경 모듈 컴파일 체크

PR 변경 파일에서 모듈 경로를 추출하고, 모듈별 빌드 타입(Android vs 순수 Kotlin)에 따라 컴파일 태스크를 결정.

# PR 변경 파일에서 모듈 경로 추출
MODULE_PATHS=$(echo "$CHANGED_FILES" \
  | grep -oP '^.+(?=/src/)' \
  | sort -u) || true

# 모듈별 컴파일 태스크 결정
while IFS= read -r MODULE_PATH; do
    GRADLE_MODULE=$(echo "$MODULE_PATH" | sed 's|/|:|g; s/^/:/')
    BUILD_FILE="${MODULE_PATH}/build.gradle.kts"

    if grep -qE 'com\.kurly\.android|com\.android' "$BUILD_FILE"; then
        TASKS+=("${GRADLE_MODULE}:compileStoreReleaseUnitTestKotlin")
    else
        TASKS+=("${GRADLE_MODULE}:compileTestKotlin")
    fi
done <<< "$MODULE_PATHS"

# 병렬 컴파일
./gradlew "${TASKS[@]}" --parallel --no-daemon 2>&1 | tee /tmp/compile-output.log

3단 분류 로직

Gradle 로그 출력에서 패턴 매칭으로 실패 유형을 결정.

유형 패턴 의미
COMPILE_ERROR grep "^e: " Kotlin 컴파일 에러 (생성자, import, 타입 불일치)
TEST_FAILURE grep -E " > .+ FAILED$" 테스트 로직 실패 (assertion, expected vs actual)
BUILD_ERROR 위 두 패턴 불일치 인프라 문제 (OOM, 플러그인 오류 등)

분류 스텝 (Classify test failure)

# 컴파일 에러가 테스트 실행 중 발생했을 수 있으므로 재확인
if grep -q "^e: " /tmp/test-output.log; then
    echo "FAILURE_TYPE=COMPILE_ERROR" >> $GITHUB_ENV
    grep "^e: " /tmp/test-output.log \
      | sed "s|e: file://||;s|e: ||;s|${WORKSPACE}||" \
      | head -20 > /tmp/error-details.txt
elif grep -qE " > .+ FAILED$" /tmp/test-output.log; then
    echo "FAILURE_TYPE=TEST_FAILURE" >> $GITHUB_ENV
else
    echo "FAILURE_TYPE=BUILD_ERROR" >> $GITHUB_ENV
    tail -50 /tmp/test-output.log > /tmp/error-details.txt
fi

PR 코멘트 자동 등록

분류 결과에 따라 PR에 코멘트를 자동 등록. 에러 목록과 조치 가이드를 포함.

  • COMPILE_ERROR: "프로덕션 코드 변경으로 테스트 코드의 컴파일을 실패했습니다" + 에러 목록
  • TEST_FAILURE: "테스트 케이스 검증을 실패했습니다" + 실패한 테스트 목록
  • BUILD_ERROR: "빌드 인프라 오류가 발생했습니다" + 마지막 50줄 로그

Slack 스레드 멘션

Slack #build_android 채널에 실패 유형별 색상 코드로 메시지를 전송하고, 스레드 댓글로 PR 작성자를 멘션. 이메일 기반으로 Slack User ID를 조회하여 <@USER_ID> 형태로 멘션.

Gemini API 컴파일 에러 자동 수정 (#7326)

COMPILE_ERROR로 분류된 경우, Gemini API를 호출하여 테스트 코드의 컴파일 에러를 자동으로 수정하고 PR에 push.

트리거 조건

  • FAILURE_TYPE == 'COMPILE_ERROR'
  • SKIP_AUTO_FIX != 'true' (무한 루프 방지)

전체 흐름

  • 1에러 파일 추출 — /tmp/error-details.txt에서 에러가 발생한 .kt/.java 파일 경로를 추출 (최대 10개)
  • 2PR 브랜치 소스 읽기 — PR 브랜치로 checkout하여 에러 파일의 현재 소스를 읽음. PR diff도 함께 수집하여 "프로덕션에서 뭐가 변경됐는지" 컨텍스트 제공
  • 3Gemini API 호출 — 에러 목록 + 소스 파일 + PR diff를 프롬프트에 포함하여 gemini-2.5-flash 호출
  • 4search/replace JSON 반환 — 전체 파일이 아닌, 수정이 필요한 부분만 {"search": "...", "replace": "..."} 쌍으로 반환
  • 5적용 & push — Python 스크립트로 search/replace 적용 후 [auto-fix] 커밋 메시지로 push

핵심 설계 결정: search/replace 방식

LLM에게 전체 파일을 반환하게 하면 출력 토큰 제한으로 파일이 잘리는(truncation) 문제가 발생한다. search/replace 쌍만 반환하면 출력량이 최소화되고, 파일 손실 위험이 없다.

프롬프트 구조

# gemini-auto-fix.sh 프롬프트 핵심 구조

"""
You are a Kotlin/Android compile error fixer.
Your ONLY job is to fix compile errors with minimal, surgical changes.

## Rules
1. Fix ONLY the compile errors listed — nothing else.
2. Do NOT refactor, rename, or improve code.
3. Each replacement must be an EXACT substring match.
4. Include enough surrounding context in "search" to make it unique.
5. "replace" should be "" (empty string) for deletions.

## Compile Errors
${ERRORS}

## Recent Production Code Changes (context)
${PR_DIFF}

## Source Files With Errors
${SOURCE_CONTEXT}

## Output Format
[{
  "file": "path/to/Test.kt",
  "replacements": [
    { "search": "oldParam = oldParam,\n", "replace": "" },
    { "search": "import com.example.OldClass",
      "replace": "import com.example.NewClass" }
  ]
}]
"""

응답 파싱 및 적용

# Gemini API 호출 설정
REQUEST_BODY=$(jq -n \
  --arg text "$PROMPT" \
  '{
    contents: [{ parts: [{ text: $text }] }],
    generationConfig: {
      responseMimeType: "application/json",
      temperature: 0.1
    }
  }')

# 응답 파싱: candidates[0].content.parts[0].text
FIXES_TEXT=$(echo "$RESPONSE_BODY" \
  | jq -r '.candidates[0].content.parts[0].text // empty')

# JSON 유효성 검증 + 파일 존재 여부 필터
VALID_FIXES=$(echo "$FIXES_TEXT" \
  | jq '[.[] | select(.file != null and .replacements != null
    and (.replacements | length) > 0)]')

Search/Replace 적용 (Python)

# 워크플로우에 임베딩된 Python 스크립트
import json, os

fixes = json.load(open('/tmp/gemini-fixes.json'))
applied = 0

for fix in fixes:
    if not os.path.exists(fix['file']):
        continue
    with open(fix['file']) as f:
        content = f.read()
    for r in fix['replacements']:
        if r['search'] in content:
            content = content.replace(r['search'], r['replace'])
            applied += 1

무한 루프 방지

Gemini가 수정한 코드를 push하면 synchronize 이벤트로 CI가 다시 트리거됨. 무한 루프를 방지하기 위해 워크플로우 초기에 마지막 커밋 메시지를 확인.

# [auto-fix] 커밋 무한 루프 방지
LAST_MSG=$(curl -s \
  -H "Authorization: token ${{ secrets.PULL_REQUEST_TOKEN }}" \
  "${{ github.api_url }}/repos/.../commits/${{ github.event.pull_request.head.sha }}" \
  | jq -r '.commit.message // ""')

if [[ "$LAST_MSG" == *"[auto-fix]"* ]]; then
    echo "SKIP_AUTO_FIX=true" >> $GITHUB_ENV
fi

전체 파이프라인 흐름

3개 PR이 연결되는 전체 파이프라인 흐름을 인터랙티브 시각화로 확인할 수 있다. 각 단계별 Before/After 비교와 애니메이션이 포함되어 있다.

변경 분석 게이트 → 실패 분류 + 알림 → Gemini 자동 수정
시각화 보기 →

한계

  • Gemini 자동 수정 성공률: 생성자 파라미터 추가, import 변경 같은 단순 케이스에서는 잘 동작하지만, 복잡한 리팩토링(클래스 구조 변경, 제네릭 타입 변경 등)은 수동 수정이 필요함.
  • 분류 패턴의 Gradle 의존성: ^e:와 > .+ FAILED$ 패턴은 현재 Gradle 출력 형식에 의존. Gradle 버전 업데이트 시 출력 형식이 변경되면 분류가 깨질 수 있음. test_classifier.sh fixtures로 사전 검증 가능.
  • 간접 의존성 미감지: check-changes의 파일 경로 기반 판단은 모듈 간 간접 의존성을 고려하지 않음. 예: 모듈 A의 테스트만 수정했지만, A의 인터페이스가 B에 영향을 주는 경우는 감지 불가.
