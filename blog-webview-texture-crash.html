크래시 1만 건, 지표가 흔들리다

2025년 4분기, 상품상세 화면에서 크래시 발생 건수가 급격히 증가했다. Crashlytics 로그를 확인한 결과, Chromium WebView의 네이티브 크래시가 앱 프로세스까지 전이된 것으로 파악됐다. 프레임워크 내부에서 발생한 예외였기 때문에, 앱 단에서 try-catch로 직접 핸들링할 수 있는 이슈가 아니었다.

처음에는 크래시 추이를 모니터링하되, 추가 증가가 없으면 별도 대응 없이 유지하는 방향으로 커뮤니케이션했다. 그러나 채 얼마 지나지 않아 상황이 바뀌었다.

크래시 건수가 1만 건을 돌파하며 앱 전체 안정성 지표를 흔들었다. 프레임워크 문제로만 치부하기에는 영향 범위가 지나치게 컸다.

이 시점에서 "프레임워크 문제이니 우리가 할 수 있는 것이 없다"는 판단을 재검토하게 됐다. 근본 원인을 해결할 수 없더라도, 크래시로 이어지는 경로를 우회할 수 있는 방법이 있지 않을까?

실마리: 16383이라는 숫자

Crashlytics에는 두 가지 유형의 크래시가 보고되고 있었다. UncaughtException은 실마리를 찾기 어려웠고, 상대적으로 단서가 있는 IllegalStateException에 집중하기로 했다.

Crashlytics 로그

IllegalStateException: Unable to create layer — requested size 16384, max size 16383

WebView의 최대 레이어 크기가 16383인데, 16384 크기의 레이어를 요청했기 때문에 생성에 실패했다는 것이다. 딱 1px 차이였다.

크래시가 발생한 디바이스를 살펴보니, 주로 저사양 단말에서 집중적으로 발생하는 패턴이 확인됐다.

디바이스출시GPU
Galaxy A152024Mali-G57
Galaxy A342023Mali-G68
Galaxy S102019Mali-G76
Galaxy S212021Mali-G78
Galaxy NOTE102019Mali-G76

특이한 점이 하나 더 있었다. 사용자들은 서로 다른 상품을 탐색하고 있었는데, 크래시가 발생한 시점의 로그 값이 모두 16383이었다. WebView 컨텐츠 길이가 모두 같다는 것은 쉽게 납득하기 어려웠다.

어쩌면 로그에 찍힌 16383이라는 값이 컨텐츠 길이를 의미하는 게 아닐 수도 있겠다.

이 의문이 조사의 방향을 바꿨다.

GPU 텍스처의 경계에서 일어난 일

SkiaGpuPipeline.cpp 추적

실제로 예외가 발생하는 지점을 추적한 결과, Android HWUI 프레임워크의 SkiaGpuPipeline::createOrUpdateLayer()에서 동일한 에러 로그를 출력하고 있었다. 이 함수는 GPU에 올릴 오프스크린 레이어(Texture/FrameBuffer)가 필요할 때 호출된다.

앱 레벨에서 호출하는 setLayerType(View.LAYER_TYPE_HARDWARE)가 간접적인 트리거가 될 수 있다. RenderNode가 레이어 합성을 요구하는 경우, 별도의 단일 레이어를 생성하기 위해 이 함수를 호출하기 때문이다.

함수 내부 구현을 통해 핵심적인 사실을 확인했다. 로그의 16383은 컨텐츠 길이가 아니라 maxTextureSize였다. GPU가 생성할 수 있는 최대 텍스처 크기를 의미한다.

디바이스별 maxTextureSize

그렇다면 실제로 디바이스마다 maxTextureSize가 어떻게 다른지 확인해보았다. OpenGL ES의 GL_MAX_TEXTURE_SIZE 값을 EGL 컨텍스트를 통해 직접 조회했다.

디바이스maxTextureSize64로 나눈 나머지위험도
Pixel 10 32768 0 안전
Galaxy Note20, Z Flip5 16384 0 안전
Galaxy A24, A25, S21 16383 63 위험

32768이나 16384처럼 64의 배수인 디바이스는 문제가 없고, 16383처럼 64의 배수가 아닌 디바이스에서만 크래시가 발생하고 있었다. 왜 64가 중요한 것일까?

LAYER_SIZE: 64배수 올림의 함정

Android HWUI 프레임워크의 소스 코드에 답이 있었다.

// frameworks/base/libs/hwui/Properties.h
// Textures used by layers must have dimensions multiples of this number
#define LAYER_SIZE 64

GPU 레이어에서 사용되는 텍스처의 크기는 반드시 64의 배수여야 한다. 그래서 레이어를 생성할 때 입력된 높이를 64의 배수로 올림하여 실제 Surface 크기를 결정한다.

// SkiaGpuPipeline.cpp — Surface 높이 계산
const int surfaceHeight = ceilf(height / float(LAYER_SIZE)) * LAYER_SIZE;

이 계산 로직의 결과를 구체적으로 살펴보면 경계 문제가 명확해진다.

LAYER_SIZE 64배수 올림 계산
16319 ⌈16319 ÷ 64⌉ × 64 = 255 × 64 = 16320 ✓
16320 ⌈16320 ÷ 64⌉ × 64 = 255 × 64 = 16320 ✓
16383 ⌈16383 ÷ 64⌉ × 64 = 256 × 64 = 16384 ✗
16384 ⌈16384 ÷ 64⌉ × 64 = 256 × 64 = 16384 ✗

maxTextureSize = 16383인 디바이스에서 높이가 16321~16383 범위에 들어가면, 올림 결과가 16384가 되어 maxTextureSize를 초과한다. 입력 값 자체는 한계 이내이지만, 정렬 과정에서 한계를 넘는다.

핵심 요약

maxTextureSize가 64의 배수가 아닌 디바이스에서, WebView 높이가 특정 구간(16321~16383)에 진입하면 LAYER_SIZE 올림으로 인해 텍스처 크기가 한계를 초과한다. 이것이 네이티브 크래시의 직접적 원인이었다.

가설 검증: 크래시를 재현하기까지

원인을 파악했지만, 크래시를 직접 재현해야 확신할 수 있었다. 크래시를 수집한 지 3개월 만에 QA팀으로부터 저사양 단말(Galaxy A25, maxTextureSize = 16383)을 대여했다. 하지만 동일한 상품을 탐색해도 문제는 재현되지 않았다.

SkiaGpuPipeline::createOrUpdateLayer()가 호출되지 않으면 크래시는 재현할 수 없다. 그렇다면 이 함수를 의도적으로 트리거해볼 수 있는 방법은 없을까?

실험 1: 투명도와 렌더링 전략

WebView에 투명도를 적용하면, 타일(Tile) 기반이 아니라 컨텐츠 전체 영역을 한 번에 렌더링하는 동작으로 바뀔 수 있다는 가설을 세웠다. 만약 그렇다면 GPU가 별도의 레이어를 생성하며 합성할 것이고, 이때 createOrUpdateLayer()가 호출될 것이다.

컨텐츠 ≤ maxTextureSize

높이 13,216px + 투명도 적용

투명도 정상 적용됨
전체 영역 단일 렌더링 (Chrome Inspector 확인)

컨텐츠 > maxTextureSize

높이 35,177px + 투명도 적용

투명도 미적용됨
타일 기반 분할 렌더링으로 전환

결과는 명확했다. maxTextureSize 이하의 WebView는 전체를 한 번에 렌더링하고, 초과하면 타일 기반으로 전환한다. 하지만 기대와 달리, 이 실험에서는 크래시가 재현되지 않았다.

실험 2: WebView 높이 고정

크래시가 발생한 상품들의 실제 WebView 컨텐츠 길이는 대부분 위험 구간(16321~16383)에 들어가지 않았다. 그래서 WebView 높이를 의도적으로 위험 구간에 진입시켜보았다.

binding.webView.updateHeight(16381) // 16321~16384 위험 구간
동일한 IllegalStateException이 발생하며 앱이 종료됐다. wrap_content로 설정된 WebView가 특정 조건에서 위험 구간에 진입하면 크래시를 유발할 수 있다는 것을 재현을 통해 확인했다.

LAYER_TYPE_NONE으로 우회하다

실험에서 한 가지 추가 발견이 있었다. View.LAYER_TYPE_HARDWARE 대신 View.LAYER_TYPE_NONE으로 변경했을 때, 컨텐츠 길이와 무관하게 항상 타일 기반 렌더링이 유지됐다.

LAYER_TYPE_HARDWARE
  • maxTextureSize 이하 → 단일 레이어 합성
  • maxTextureSize 초과 → 타일 기반 분할
  • 경계 구간에서 크래시 위험
LAYER_TYPE_NONE
  • 별도 레이어를 강제하지 않음
  • Chromium 기본 타일 렌더링 경로 유지
  • 경계 구간 크래시 회피

즉, LAYER_TYPE_HARDWARE가 특정 조건에서 WebView의 렌더링 전략을 타일 기반에서 단일 레이어 기반으로 바꿀 수 있고, 이때 텍스처 크기 경계 문제가 발생하는 것으로 추정된다. LAYER_TYPE_NONE으로 바꾸면 이 경로를 우회할 수 있다.

LayerTypeInspector 구현

모든 디바이스에서 일괄적으로 LAYER_TYPE_NONE을 적용하는 것은 과한 대응이다. maxTextureSize가 64의 배수인 디바이스는 이 문제에 해당하지 않기 때문이다. 디바이스의 GPU 능력을 런타임에 조회하여 레이어 타입을 결정하는 유틸리티를 구현했다.

object LayerTypeInspector {

    val maxTextureSize: Int by lazy {
        queryMaxTextureSize() ?: 0
    }

    private const val LAYER_SIZE = 64

    fun getType(enableHardwareLayer: Boolean): Int {
        return if (maxTextureSize % LAYER_SIZE == 0) {
            View.LAYER_TYPE_HARDWARE
        } else {
            when {
                enableHardwareLayer -> View.LAYER_TYPE_HARDWARE
                else -> View.LAYER_TYPE_NONE
            }
        }
    }
}

핵심 판단 로직은 간단하다. maxTextureSize % 64 == 0이면 안전하므로 HARDWARE를 유지하고, 그렇지 않으면 NONE으로 우회한다.

queryMaxTextureSize()는 EGL 컨텍스트를 생성하여 GL_MAX_TEXTURE_SIZE를 조회한다. lazy로 한 번만 실행되며, 실패 시 0을 반환하여 NONE 경로로 진입한다.

AB 테스트로 안전망 확보

이번 대응은 추측성 방어코드이므로, 이슈가 해소되지 않으면 롤백할 수 있어야 했다. 하드웨어 레이어 활성화 여부를 원격으로 제어할 수 있는 AB 테스트 Feature를 추가했다.

@Singleton
@ABTestFeature
class HardwareLayerFeature @Inject constructor()
    : Feature<HardwareLayerFeature.Value>() {

    override val key = FeatureKeys.HARDWARE_LAYER_ENABLED

    sealed class Value(override val rawValue: Boolean)
        : FeatureBooleanValue {
        data object Enable : Value(true)
        data object Disable : Value(false)
    }
}

상품상세 화면에서 이 값을 조회하여 LayerTypeInspector.getType()에 전달한다. AB 테스트가 Enable이면 기존 HARDWARE를 강제하고, Disable이면 LayerTypeInspector의 판단에 따른다.

private fun enableHardwareLayer(): Boolean {
    return abTestManager
        .getValue<HardwareLayerFeature.Value>(
            FeatureKeys.HARDWARE_LAYER_ENABLED
        )
        ?.rawValue
        ?: true // 기본값: 기존 동작 유지
}

기본값을 true(기존 동작 유지)로 설정하여, AB 테스트를 명시적으로 활성화하기 전까지는 기존 동작에 영향을 주지 않도록 했다.

한계와 향후 과제

이번 대응에는 명확한 한계가 있다.

  • 근본 해결이 아닌 우회 — 프레임워크/Chromium 레벨의 문제를 앱에서 회피한 것이다. WebView 업데이트로 근본 이슈가 해소되는 시점에는 우회 로직을 완화하거나 제거해야 한다.
  • LAYER_TYPE_NONE의 부작용 가능성 — LAYER_TYPE_HARDWARE는 하드웨어 가속 렌더링 레이어를 강제하여 특정 상황에서 성능 이점을 제공한다. NONE으로 전환하면 이 이점을 포기하게 된다. 다만, 문제가 되는 디바이스는 대부분 저사양 단말이므로 하드웨어 레이어 합성의 이점보다 안정성이 더 중요하다고 판단했다.
  • wrap_content의 구조적 문제 — 상품상세 WebView가 wrap_content를 사용하고 있어, 컨텐츠 길이에 따라 런타임에 높이가 크게 변동된다. 위험 구간(16321~16383)으로 진입할 확률은 낮지만 0이 아니다. match_parent 또는 고정 높이 정책으로 전환하는 것이 바람직하다.

Takeaways

  • 1 "프레임워크 문제"가 곧 "할 수 있는 게 없다"는 뜻은 아니다. 근본 원인을 해결할 수 없어도 크래시로 이어지는 경로를 차단할 수 있다. 이번 사례에서는 렌더링 전략을 바꾸는 것만으로 크래시 경로를 우회했다.
  • 2 동일한 숫자가 반복되면 그 숫자의 의미를 의심하라. 16383이 "컨텐츠 길이"라는 초기 가정에서 "maxTextureSize"라는 올바른 해석으로 전환한 것이 조사의 전환점이었다.
  • 3 정렬(alignment)은 "보이지 않는 +1"을 만든다. 메모리 정렬, 텍스처 정렬, 페이지 정렬 — 정렬 연산은 입력 값이 경계에 가까울 때 예상치 못한 초과를 유발한다. 경계 근처에서 동작하는 시스템은 정렬 단위까지 고려해야 한다.
  • 4 추측성 방어코드에는 항상 롤백 경로를 함께 구현하라. AB 테스트 플래그를 통해 원격으로 동작을 제어할 수 있게 한 것이 안전망이 됐다. 방어코드가 새로운 문제를 만들면 즉시 원복할 수 있어야 한다.

WebView를 사용하는 앱이라면, 저사양 디바이스의 GL_MAX_TEXTURE_SIZE를 한 번쯤 조회해볼 것을 권한다. 지금 아무 문제가 없더라도, 컨텐츠 길이가 조금만 바뀌면 동일한 경계에 진입할 수 있다.
