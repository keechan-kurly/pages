배경

컬리 앱의 메인화면에는 7개의 팝업이 순차적으로 노출됩니다. 권한 안내, 알림 권한, 멤버스 전환 동의, 마케팅 수신 동의, 바텀시트, 공지사항, 비밀번호 변경 안내 -- 이 팝업들은 이전 팝업의 결과에 따라 다음 팝업의 노출 여부가 결정되는 순차적 의존 관계를 가지고 있었습니다.

기존 구현에서는 이 모든 로직이 MainActivity와 MainViewModel 두 클래스에 집중되어 있었습니다.

순서 변경 위험
Activity(174줄) + ViewModel(191줄)을 동시에 수정해야 팝업 순서를 변경할 수 있음. 한쪽만 수정하면 동작이 깨짐.
5곳 수정 필요
새 팝업 추가 시 ViewModel의 UIEvent, check 함수, Activity의 show 함수, when 분기, 순서 로직을 모두 수정.
테스트 불가
팝업 순서 검증은 Activity와 ViewModel의 UIEvent 수집에 의존. 단위 테스트로 순서를 검증할 수 없음.

예를 들어, 기존 MainViewModel에서 마케팅 팝업 표시 후 딥링크를 체크하는 코드는 다음과 같았습니다.

// MainViewModel.kt (기존 코드 -- 226줄 중 일부)
fun checkMarketingPopup() {
    viewModelScope.launch(dispatcher.io()) {
        when (checkMarketingPopup.execute()) {
            is CheckMarketingPopup.Result.Show -> {
                isShownMarketingPopup = true
                _uiEvent.emit(UIEvent.ShowMarketingPopup)
            }
            is CheckMarketingPopup.Result.Pass -> {
                _uiEvent.emit(UIEvent.CheckDeepLink)  // 다음 단계를 직접 지정
            }
        }
    }
}

각 팝업 함수가 다음 팝업 함수를 직접 호출하는 구조였기 때문에, 팝업 간의 순서가 코드 전체에 흩어져 있었습니다.

1차 개선 -- Chain of Responsibility (#5669)

요구사항과 설계

요구사항설계
팝업 단위 독립 PopupHandler interface + BasePopupHandler
순서를 한 곳에서 관리 MainPopupFlow -- handler chaining
핸들러 간 통신 SideEffect sealed interface
딥링크 시 조건 분기 NopPopupHandler -- 조건에 따라 스킵
팝업 순서 검증 MainPopupFlowTest + PopupHandlerTest

PopupHandler interface

각 팝업을 독립된 핸들러로 추출하기 위한 인터페이스입니다. next()로 체이닝하고, launch()로 실행합니다.

interface PopupHandler {
    fun next(block: () -> PopupHandler): PopupHandler
    fun next(handler: PopupHandler): PopupHandler
    fun next(isShown: Boolean)
    fun launch(
        previousPopup: PopupHandler = this,
        isShown: Boolean = false
    )
    fun bind(executorService: PopupExecutorService): PopupHandler
}

next(block: () -> PopupHandler) 오버로드는 런타임 조건 분기를 가능하게 합니다. 딥링크 여부에 따라 NopPopupHandler 또는 MarketingBottomSheetPopupHandler를 동적으로 결정합니다.

MainPopupFlow -- 순서를 한 곳에서 관리

7개 핸들러의 실행 순서가 단 하나의 체이닝 표현식으로 정의됩니다.

class MainPopupFlow(
    private val isHandledDeepLink: () -> Boolean,
    private val permissionUsageGuidePopupHandler: PopupHandler,
    private val notificationPermissionPopupHandler: PopupHandler,
    // ... 5개 핸들러 생략
    private val popupCompletionExecutor: PopupCompletionExecutor,
    private val popupTaskExecutor: PopupTaskExecutor
) {
    private val popupHandler: PopupHandler by lazy {
        permissionUsageGuidePopupHandler
            .next(notificationPermissionPopupHandler)
            .next(membersSubscriptionAgreePopupHandler)
            .next(marketingPopupHandler)
            .next(getNextHandler)  // 딥링크 조건 분기
            .next(noticePopupHandler)
            .next(passwordChangePopupHandler)
            .bind(executorService)
    }

    private val getNextHandler = {
        if (isHandledDeepLink()) nopPopupHandler
        else marketingBottomSheetPopupHandler
    }
}

SideEffect -- 핸들러와 Activity 간 통신

핸들러가 Activity의 UI 동작(Snackbar, 이벤트 전송 등)을 요청할 때 사용하는 sealed interface입니다.

sealed interface SideEffect {
    sealed class MarketingPopup : SideEffect {
        data class ShowSnackbar(val message: String) : MarketingPopup()
        data class SendMarketingInformationAgreement(val isChecked: Boolean) : MarketingPopup()
        data object BlockViewMainBrazeEventWhileDeepLinkMoving : MarketingPopup()
    }
    sealed interface NoticePopup : SideEffect { /* ... */ }
    sealed class PasswordPopup : SideEffect { /* ... */ }
}

문제 발생 -- Lifecycle 크래시

PR #5669를 배포한 후, FragmentTransaction 관련 크래시가 보고되었습니다.

IllegalStateException: Can not perform this action after onSaveInstanceState

PopupHandler가 Activity의 Lifecycle을 인지하지 못한 채 FragmentTransaction을 시도하면, Activity가 onSaveInstanceState 이후 상태(STOPPED)일 때 트랜잭션이 실패합니다.

원인을 분석하면 명확합니다.

  • 각 핸들러의 launch()는 coroutineScope.launch { }로 비동기 실행됨
  • API 호출 후 DialogFragment를 show()하는 시점에 Activity가 이미 백그라운드로 전환되었을 수 있음
  • 기존 구조에서는 핸들러가 Lifecycle 상태를 확인하는 메커니즘이 없었음

추가로, 핸들러 내 코루틴에서 예외가 발생하면 체인이 끊어져 이후 팝업이 영원히 실행되지 않는 문제도 있었습니다.

2차 개선 -- Lifecycle 기반 구조 (#5778)

요구사항과 설계

요구사항설계
Lifecycle이 필요한 작업을 안전하게 실행 PopupTask + PopupTaskExecutor
RESUMED 상태에서만 FragmentTransaction repeatOnResumed -- Channel 기반 태스크 큐
예외 발생 시 다음 핸들러로 자동 전환 PopupHandlerInterceptor + CoroutineExceptionHandler
팝업 플로우 완료 알림 PopupCompletionExecutor -- SharedFlow delegate
3가지 executor 통합 관리 PopupExecutorService -- sideEffect + completion + task

PopupTask -- Lifecycle 의존 작업 추상화

FragmentTransaction 등 Lifecycle이 필요한 작업을 fun interface로 추출합니다.

fun interface PopupTask {
    fun execute()
}

PopupTaskExecutor -- RESUMED 상태에서만 실행

핵심 컴포넌트입니다. Channel로 태스크를 큐잉하고, repeatOnResumed로 Activity가 RESUMED 상태일 때만 소비합니다.

class PopupTaskExecutor(
    lifecycleOwner: LifecycleOwner,
    channel: Channel<PopupTask>
) : Channel<PopupTask> by channel {

    init {
        lifecycleOwner.repeatOnResumed {
            receiveAsFlow().collect {
                it.execute()
            }
        }
    }

    companion object {
        fun new(lifecycleOwner: LifecycleOwner): PopupTaskExecutor {
            return PopupTaskExecutor(lifecycleOwner, Channel())
        }
    }
}

이 구조에서 핸들러가 submit(task)를 호출하면 Channel에 태스크가 적재되고, Activity가 RESUMED 상태가 되어야 비로소 execute()가 호출됩니다. 백그라운드 전환 후 복귀했을 때 자연스럽게 재개됩니다.

PopupExecutorService -- 3가지 경로 통합

핸들러가 사용하는 3가지 실행 경로(SideEffect 전달, 완료 알림, 태스크 제출)를 하나의 서비스로 통합합니다.

data class PopupExecutorService(
    private val sideEffectExecutor: (SideEffect) -> Unit,
    private val completionExecutor: () -> Unit,
    private val taskExecutor: suspend (PopupTask) -> Unit
) {
    fun sideEffect(sideEffect: SideEffect) { sideEffectExecutor(sideEffect) }
    fun complete() { completionExecutor() }
    suspend fun submit(task: PopupTask) { taskExecutor(task) }

    companion object {
        val Default = PopupExecutorService({}, {}, {})
    }
}

PopupHandlerInterceptor -- 예외 시 체인 복구

CoroutineContext.Element를 활용하여 핸들러를 코루틴 컨텍스트에 포함시킵니다. CoroutineExceptionHandler에서 현재 핸들러를 꺼내 next()를 호출함으로써 예외가 발생해도 체인이 끊어지지 않습니다.

class PopupHandlerInterceptor(
    private val popupHandler: PopupHandler
) : CoroutineContext.Element {

    override val key: CoroutineContext.Key<*> = Key

    fun next() { popupHandler.next(false) }

    companion object Key : CoroutineContext.Key<PopupHandlerInterceptor>
}

DI 모듈에서 CoroutineExceptionHandler를 Activity 스코프로 제공합니다.

@Provides
@ActivityScope
fun provideCoroutineExceptionHandler(logger: Logger): CoroutineExceptionHandler {
    return CoroutineExceptionHandler { coroutineContext, throwable ->
        logger.error(throwable)
        coroutineContext[PopupHandlerInterceptor]?.next()
    }
}

핸들러에 적용된 모습

각 핸들러는 FragmentTransaction을 PopupTask로 래핑하여 submit()으로 전달하고, 코루틴에 CoroutineExceptionHandler + PopupHandlerInterceptor를 결합합니다.

class PermissionUsageGuidePopupHandler @Inject constructor(
    @ActivityScope private val coroutineScope: CoroutineScope,
    @ActivityScope private val coroutineExceptionHandler: CoroutineExceptionHandler,
    private val popupDispatcher: PermissionUsageGuidePopupDispatcher,
    private val checkPermissionPopup: CheckPermissionPopup
) : BasePopupHandler(coroutineScope) {

    override fun launch(previousPopup: PopupHandler, isShown: Boolean) {
        coroutineScope.launch(coroutineExceptionHandler + createPopupHandlerInterceptor()) {
            when (checkPermissionPopup.execute()) {
                is CheckPermissionPopup.Result.Show -> {
                    val task = PopupTask {
                        popupDispatcher.open(BasicPopup { next(true) })
                    }
                    submit(task)  // RESUMED에서만 실행됨
                }
                else -> next(false)
            }
        }
    }
}

3차 개선 — 테스트 커버리지 100% (#5853)

#5778의 Lifecycle 구조 도입 후, 각 핸들러의 UI 로직을 PopupDispatcher로 분리하여 테스트 가능성을 높이고 커버리지를 100%로 달성.

핵심 변경: Handler → Dispatcher 분리

Before

핸들러 안에 UseCase 호출 + FragmentTransaction + 콜백 로직이 혼재
→ 테스트에서 Fragment 의존성 필요

After

핸들러는 조건 판단만, UI 로직은 PopupDispatcher로 위임
→ Dispatcher를 mock하여 핸들러 단독 테스트 가능

구조 비교

Before -- #5669 only

Chain of Responsibility 도입

  • God Object 분리 완료
  • 팝업 순서 테스트 가능
  • SideEffect로 Activity 통신

잔존 문제: FragmentTransaction 타이밍 제어 불가. 예외 시 체인 끊김.

After -- #5669 + #5778

Lifecycle 기반 안정성 추가

  • PopupTask + PopupTaskExecutor로 RESUMED 보장
  • PopupHandlerInterceptor로 예외 복구
  • PopupExecutorService로 실행 경로 통합

결과: Lifecycle 크래시 0건, 체인 중단 0건.

한계

  • PopupTaskExecutor의 Channel 무한 대기 -- Channel()은 버퍼 없는 rendezvous channel입니다. 태스크를 send한 뒤 Activity가 RESUMED로 돌아오지 않으면(예: 프로세스 종료) 코루틴이 무한 대기합니다. timeout 메커니즘이 없습니다.
  • PopupHandlerInterceptor의 테스트 복잡도 -- CoroutineContext.Element로 핸들러를 컨텍스트에 주입하는 구조는, 테스트 시 커스텀 CoroutineContext를 구성해야 하므로 단위 테스트 설정이 다소 복잡해집니다.
핵심 요약
  • 1차 (#5669): Chain of Responsibility 패턴으로 God Object(MainActivity 174줄 + MainViewModel 191줄)를 7개 독립 핸들러 + 1개 Flow 클래스로 분리. 팝업 순서 테스트 가능.
  • 2차 (#5778): PopupTask + PopupTaskExecutor로 RESUMED 상태에서만 FragmentTransaction 실행. CoroutineExceptionHandler + PopupHandlerInterceptor로 예외 시 체인 자동 복구.
