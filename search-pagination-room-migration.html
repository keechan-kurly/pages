검색 Pagination 캐시를
Room DB로 마이그레이션

앱이 죽어도 검색 결과가 살아남는 구조로 — 메모리 캐시의 3가지 한계를 Room 단일 진실 소스로 해결

사용자 경험
프로세스 재생성 후에도 검색 결과 유지 — 재검색 불필요
스크롤 성능
O(n) → O(1) position 조회 + peek() 페이지 로드 방지
유지보수성
Mapper 변경해도 캐시 무효화 불필요 — raw JSON 분리 저장
코드 복잡도
ViewHolder 11개 bind() 시그니처 단순화 — position 파라미터 제거
1
캐시 영속화
Journey
검색 페이지네이션 사이사이에 연관 상품·배너·구매 이력 등을 끼워 넣어야 했다. 메모리 캐시로 이 삽입을 관리하면 부수 정보가 누락되기 쉽고, 실제로 비정상 동작이 발생했다 → Room에 위임하여 SQL이 책임지도록 전환.
메모리 캐시의 한계
✗
페이지 사이에 아이템을 INSERT하면 페이지 경계 인덱스, 위치 보정, 메타정보를 앱 코드에서 직접 관리해야 함
✗
부수 정보 관리를 놓치면 삽입된 섹션의 위치가 부정확 → 실제로 일부 케이스에서 비정상 동작 발생
✗
앱이 백그라운드에서 종료되면 HashMap 전체가 소멸 → 사용자가 처음부터 재검색
Room DB로 위임한 결과
✓
각 섹션을 Row로 저장, ORDER BY orderKey로 정렬 → SQL이 삽입·정렬을 책임
✓
sectionId, sectionPosition, sectionArea가 Entity 컬럼 → 메타데이터 누락 자체가 불가능한 구조
✓
디스크에 저장되므로 앱이 종료되어도 캐시가 생존 — 재검색 불필요
Before: 삽입·정렬·메타정보를 앱 코드가 직접 관리 → 누락 시 버그
After: 삽입·정렬·메타정보를 Room + SQL이 책임 → 구조적으로 누락 불가
2
변환 시점 최적화
Flow
API 응답을 받자마자 도메인 모델로 변환하면, 원본 JSON이 사라져서 Room에 저장할 수 없다 → 원본을 먼저 저장하고, 화면에 보여줄 때 변환하는 2단계로 분리.
Before — 변환이 한 곳에서 즉시 일어남
1
서버에서 검색 결과 JSON 수신
⚡
TypeAdapter가 즉시 도메인 모델로 변환
JSON → SearchPagingModel (sealed class 포함) — 이 시점에서 원본 JSON이 사라짐
✗
메모리 캐시에 도메인 모델을 저장
원본 JSON 유실 → Room 저장 불가 (sealed class 직렬화 복잡) / Mapper 변경 시 캐시 전체 무효화
After — 저장과 변환을 2단계로 분리
Stage 1 — 저장 시점
1
서버에서 검색 결과 JSON 수신
2
TypeAdapter가 메타데이터만 추출
orderKey, sectionId, sectionCode — 도메인 모델로 변환하지 않음
3
Room에 raw JSON + 메타데이터를 그대로 저장
원본 보존 → sealed class 직렬화 불필요 / 캐시 무효화 불필요
Stage 2 — 화면에 보여줄 때
4
Room PagingSource가 Entity를 읽어옴
5
이 시점에 Mapper가 도메인 모델로 변환
Mapper 로직이 바뀌어도 Room의 캐시는 그대로 — 재파싱만 하면 됨
3
UI 바인딩 단순화
Journey
"이 아이템이 몇 번째 섹션의 몇 번째인지"를 화면에 그릴 때마다 전체 목록을 뒤져서 계산했다 → 데이터를 저장할 때 위치를 확정하여 화면은 받아서 그리기만 하도록 단순화.
화면이 너무 많은 일을 했다
✗
리스트를 그릴 때마다 "이 아이템이 몇 번째인지"를 전체 목록에서 매번 탐색
snapshot().filter{}.groupBy{}.findIndex{}
✗
아이템 ID를 10개 이상의 Mapper가 제각각 생성 → 어디서 만든 ID인지 추적 불가
✗
DiffUtil이 hashCode() 비교 → 상태만 바뀌어도 "다른 아이템"으로 판정 → 전체 교체 발생
✗
ViewType 확인에 getItem() 사용 → 보기만 해도 다음 페이지를 프리페치
저장할 때 다 정해두면 화면은 그리기만
✓
API 파싱 시점에 위치를 한 번만 계산하여 Entity에 저장 → 화면은 꺼내 쓰기만 (O(1))
✓
ID를 TypeAdapter에서 한 곳에서 한 번만 생성 → 모든 Mapper가 동일 ID 참조
✓
DiffUtil이 stable id 비교 → 상태 변경 시 해당 아이템만 갱신, 나머지는 유지
✓
peek()으로 이미 로드된 데이터만 참조 → 불필요한 페이지 로드 제거
Before: bind(item, getSectionPosition(item), getProductPosition(position))
After: bind(item) — item이 이미 자신의 위치를 알고 있다
Change Summary
  • 삭제 SearchProductCachePool (148줄) + SearchProductCacheSource (37줄) — 메모리 캐시 전체 제거
  • 추가 SearchSectionEntity + SearchPageKeyEntity + SearchDao + SearchProductTransaction — Room 기반 캐시 레이어
  • 추가 SearchSectionAreaTypeAdapter + SerializationSnapshot — sealed class 다형성 Room 저장
  • 전환 Eager 도메인 변환 → Lazy raw JSON 저장 + UI 바인딩 시점 변환
